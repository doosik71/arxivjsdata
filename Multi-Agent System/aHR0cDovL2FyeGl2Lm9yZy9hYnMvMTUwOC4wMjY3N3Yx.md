# Call Graph Profiling for Multi Agent Systems

Dinh Doan Van Bien, David Lillis and Rem W. Collier (2015)

## 🧩 Problem to Solve

본 논문이 해결하고자 하는 핵심 문제는 Multi Agent System (MAS) 개발 과정에서 시스템의 성능 분석 및 효율성 개선을 위한 전문적인 프로파일링(Profiling) 도구가 부족하다는 점이다. MAS는 느슨하게 결합된 자율적인 소프트웨어 엔티티들이 상호 작용하는 복잡한 구조를 가지므로, 기존의 객체 지향 프로그래밍이나 순차적 프로그램 분석 도구로는 그 특성을 완전히 파악하기 어렵다.

전통적인 프로파일링은 프로그램 실행 시간의 대부분을 차지하는 소수의 '병목 지점(Bottleneck)' 또는 '핫스팟(Hot spot)'을 찾아내어 이를 최적화함으로써 전체 성능을 향상시키는 기법이다. 그러나 MAS에서는 단순히 어떤 함수가 오래 걸렸는가보다, **에이전트 간의 통신(Communication)이 시스템 전체의 처리 요구량에 어떤 영향을 미쳤는가**를 파악하는 것이 훨씬 중요하다. 따라서 본 논문의 목표는 에이전트 지향 프로그래밍의 개념을 반영하여, 통신 기반의 성능 분석이 가능한 'Call Graph View'를 AgentSpotter 도구에 도입하는 것이다.

## ✨ Key Contributions

본 논문의 핵심 기여는 전통적인 소프트웨어 공학의 Call Graph 개념을 MAS 환경으로 확장하여, **에이전트 간 메시지 교환을 통한 성능 영향력을 정량화하고 이를 계층적으로 시각화**한 점이다.

중심적인 설계 아이디어는 함수 호출(Function Call)에 의한 실행 시간을 측정하는 대신, **메시지 영향력(Message Impact)**이라는 개념을 도입하는 것이다. 즉, 특정 에이전트가 메시지를 수신한 시점부터 다음 메시지를 수신하기 전까지 수행한 모든 계산 시간을 해당 메시지의 영향력으로 정의함으로써, 어떤 에이전트의 어떤 메시지가 시스템의 병목을 유발하는지를 추적할 수 있게 하였다.

## 📎 Related Works

논문에서는 기존의 프로파일링 및 디버깅 도구들을 다음과 같이 분석하며 차별점을 제시한다.

1.  **전통적 프로파일러**: `gprof`와 같은 도구는 함수 호출 스택과 실행 시간을 분석하여 계층적 프로파일을 제공하지만, MAS의 분산된 통신 구조를 분석할 수 없다.
2.  **MAS 프레임워크 도구**: `Cougaar`나 `LS/TS` 플랫폼은 시스템 모니터링 데이터를 제공하지만, 원시 데이터(Raw data)를 수집하는 수준에 그치며 프로파일러처럼 의미 있는 합성 지표(Synthetic indicators)를 생성하는 사후 처리 과정이 부족하다.
3.  **MAS 디버거 및 통신 분석 도구**: `Agent Factory Debugger`, `Brahms Agent Viewer`, `JADE Sniffer Agent` 등은 에이전트의 정신 상태나 ACL(Agent Communication Language) 메시지의 흐름을 시각화하여 논리적 오류를 찾는 데 집중한다.

본 연구는 이러한 기존 도구들이 '디버깅'이나 '단순 모니터링'에 치중된 것과 달리, **'성능 병목 지점 식별'**이라는 프로파일링 본연의 목적을 MAS의 통신 모델에 결합했다는 점에서 차별성을 가진다.

## 🛠️ Methodology

### 전체 시스템 구조 (AgentSpotter Architecture)
AgentSpotter는 플랫폼 독립적인 구조를 지향하며 다음과 같은 파이프라인을 가진다.
- **AgentSpotter Service**: 에이전트 플랫폼의 런타임 환경에서 동작하며, 에이전트의 행동, 교환된 메시지, CPU 및 메모리 사용량 등의 데이터를 수집한다.
- **Snapshot File (`.aspot`)**: 수집된 데이터를 저장하여 오프라인 분석이 가능하게 한다.
- **AgentSpotter Station**: 저장된 파일을 읽어 Query Engine을 통해 분석하고, Space-Time Diagram이나 Call Graph Tree View와 같은 시각화 결과물을 제공한다.

### Agent-Oriented Call Graph 모델 및 방정식
전통적인 Call Graph가 함수 실행 시간을 측정한다면, 본 모델은 **에이전트 메시지 영향력(Agent Message Impact)**을 측정한다.

1.  **단일 메시지 영향력**: 에이전트 $A$가 에이전트 $B$에게 시간 $\alpha$에 메시지 $M_\alpha$를 보냈을 때, $B$가 다음 메시지 $M_\Omega$를 수신하는 시간 $\Omega$까지 수행한 계산 시간 $b_t$의 합으로 정의한다.
    $$T_{M_{\alpha,B}} = \sum_{t=\alpha}^{\Omega} b_t$$

2.  **에이전트 간 총 영향력**: 에이전트 $x$가 에이전트 $y$에게 보낸 모든 메시지($M$개)의 영향력 합계이다.
    $$T_{x,y} = \sum_{m=1}^{M} \sum_{t=\alpha_m}^{\Omega_m} b_t$$

3.  **에이전트 개별 총 영향력**: 에이전트 $x$가 다른 모든 에이전트($N$명)에게 미친 영향력의 합이다.
    $$T_x = \sum_{a=1}^{N} \sum_{m=1}^{M_a} \sum_{t=\alpha_m}^{\Omega_m} b_t$$

4.  **세션 총 영향력**: 세션 $S$ 내의 모든 에이전트($K$명)가 생성한 총 영향력의 합이다.
    $$T_S = \sum_{k=1}^{K} \sum_{a=1}^{N_k} \sum_{m=1}^{M_a} \sum_{t=\alpha_m}^{\Omega_m} b_t$$

### 시각화 사양 (Call Graph Tree View)
전통적인 Call Graph는 호출 깊이에 따라 트리의 깊이가 결정되지만, 본 논문의 모델은 고정된 깊이의 계층 구조를 가진다.
- **계층 구조**: $\text{Session} \rightarrow \text{Emitter Agent} \rightarrow \text{Receiver Agent} \rightarrow \text{FIPA ACL (Performative)} \rightarrow \text{Message}$
- **표시 항목**: 각 노드에는 레이블, 총 영향 시간(Total impact time), 부모 노드 대비 비율(% parent time), 세션 대비 비율(% session time)이 표시된다.
- **유연성**: 사용자가 계층 구조를 변경(예: 수신자를 송신자보다 상위에 배치)하여 부하가 집중된 에이전트를 쉽게 식별할 수 있도록 설계되었다.

## 📊 Results

### 실험 설정
- **벤치마크 애플리케이션**: Overseer 에이전트(작업 요청자)와 Worker 에이전트(작업 수행자)로 구성된 시스템이다.
- **작업 특성**: 작업 규모(small, medium, large)에 따라 처리 시간이 달라지며, Worker가 과부하 상태일 경우 요청을 거절할 수 있다.

### 주요 결과 및 분석
1.  **부하 불균형 식별**: Call Graph 분석 결과, `master1`과 `master2` 두 Overseer 에이전트의 영향력이 서로 다르게 나타났다. `master2`의 영향력은 전체 세션의 20.4%에 불과했다.
2.  **원인 분석**: 상세 트리를 확인한 결과, `master2`가 보낸 `pleaseDoThing(20)` 메시지들이 Worker 에이전트에 의해 무시(Impact 0.0)되었음을 발견했다. 이는 `master1`의 요청이 먼저 처리되어 Worker가 과부하 상태가 되었기 때문으로 해석된다.
3.  **병목 지점 식별**: `pleaseDoThing(1)` 메시지보다 `pleaseDoThing(20)` 메시지의 영향력이 월등히 높게 나타났다. 만약 이러한 결과가 예상치 못한 것이라면, 해당 메시지가 트리거하는 내부 로직의 효율성을 개선해야 하는 병목 지점으로 판단할 수 있다.

## 🧠 Insights & Discussion

### 강점
- MAS의 특성인 '통신'을 성능 분석의 중심축으로 설정하여, 단순한 코드 실행 시간 측정을 넘어 시스템의 동역학적 병목을 찾을 수 있게 하였다.
- 고정 깊이의 트리 구조와 유연한 계층 변경 기능을 통해 개발자가 다양한 관점(송신자 중심 vs 수신자 중심)에서 성능 데이터를 해석할 수 있도록 하였다.

### 한계 및 비판적 해석
- **인과관계의 불확실성**: 본 모델은 '메시지 수신 후 다음 메시지 수신 전까지의 모든 활동'을 해당 메시지의 영향력으로 간주한다. 하지만 에이전트의 행동은 외부 환경의 변화(Percept)나 내부의 신념/목표 변화에 의해 발생할 수 있으므로, 메시지 수신과 실제 행동 사이에 명확한 인과관계가 증명되지 않는다.
- **플랫폼 의존성 문제**: 가장 이상적인 방법은 에이전트의 내부 추론(Reasoning) 과정을 추적하는 것이지만, 이는 에이전트 플랫폼마다 구현 방식이 달라 플랫폼 독립적인 도구를 지향하는 AgentSpotter의 목표와 상충된다.
- **대화 프로토콜의 부재**: 현재 모델은 개별 메시지를 독립적으로 처리한다. 그러나 실제 MAS에서는 '요청 $\rightarrow$ 수락 $\rightarrow$ 결과 통보'와 같은 일련의 대화(Conversation)가 하나의 작업 단위가 되므로, 단일 메시지 단위의 분석은 실제 처리 비용을 왜곡할 가능성이 있다.

## 📌 TL;DR

본 논문은 Multi Agent System의 성능 분석을 위해 통신 기반의 **Call Graph Profiling** 기법을 제안하였다. 메시지 수신 사이의 계산 시간을 '메시지 영향력'으로 정의하여 계층적 트리 구조로 시각화함으로써, 어떤 에이전트 간의 상호작용이 시스템 병목을 유발하는지 쉽게 식별할 수 있게 하였다. 비록 메시지와 행동 간의 엄격한 인과관계 증명에는 한계가 있으나, 통계적으로 유의미한 성능 지표를 제공함으로써 MAS의 효율성 개선 및 부하 불균형 해결에 기여할 수 있는 유용한 도구이다.
