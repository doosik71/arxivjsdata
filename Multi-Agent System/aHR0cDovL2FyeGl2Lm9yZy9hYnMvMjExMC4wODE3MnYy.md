# MLFC: From 10 to 50 Planners in the Multi-Agent Programming Contest

Rafael C. Cardoso, Angelo Ferrando, Fabio Papacchini, Matt Luckcuck, Sven Linker, Terry R. Payne (2021)

## 🧩 Problem to Solve

본 논문은 제15회 Multi-Agent Programming Contest (MAPC)의 'Agents Assemble' 시나리오에서 상위권 성적을 거두기 위해 MLFC 팀이 적용한 전략을 다룬다. 이 시나리오의 핵심 목표는 2D 그리드 맵 상에서 두 팀의 에이전트가 경쟁하며, 서버가 무작위로 생성하는 복잡한 블록 구조물을 조립하여 전달함으로써 통화를 획득하고 이를 최대화하는 것이다.

특히 이번 대회에서는 이전 대회(14회)와 비교하여 다음과 같은 세 가지 주요 제약 사항과 환경 변화가 추가되었다. 첫째, 맵의 경계가 없는 Circular Grid(원형 그리드)가 도입되어 맵의 끝에 도달하면 반대편 끝으로 연결되는 구조가 되었다. 둘째, 라운드별 에이전트 수가 15명, 30명, 50명으로 대폭 증가하여 시스템의 확장성(Scalability) 문제가 중요해졌다. 셋째, Taskboard라는 새로운 시설이 도입되어, 에이전트가 반드시 Taskboard 근처에서 작업을 수락하고 제출해야 하는 절차가 추가되었다. 따라서 본 연구의 목표는 이러한 환경 변화에 대응하여 효율적인 협력 및 경쟁 전략을 수립하고, 특히 에이전트 수 증가에 따른 계산 비용 문제를 해결하는 것이다.

## ✨ Key Contributions

본 논문의 핵심 기여는 확장된 시나리오에 대응하기 위한 다섯 가지 새로운 전략의 설계와 구현에 있다.

1.  **Cartography (지도 제작):** 맵의 경계가 사라진 원형 그리드 환경에서 맵의 정확한 가로, 세로 크기를 동적으로 파악하는 메커니즘을 제안한다.
2.  **Formal Verification of Map Merging (지도 병합 프로토콜의 형식 검증):** 에이전트 수가 증가함에 따라 지도 정보를 공유하고 병합하는 과정에서 발생할 수 있는 오류를 방지하기 위해 CSP(Communicating Sequential Processes) 언어와 FDR 모델 체커를 이용한 형식 검증을 수행하였다.
3.  **Plan Cache (계획 캐시):** 에이전트 수가 최대 50명으로 증가함에 따라 모든 에이전트가 매 단계 AI Planner를 호출할 때 발생하는 CPU 과부하 문제를 해결하기 위해, 문제 상황과 해결 계획을 매핑하여 저장하는 캐싱 기법을 도입하였다.
4.  **Bullies (방해 에이전트):** 상대 팀의 블록 조립을 방해하기 위해 특정 목표 지점을 순찰하며 상대의 블록을 제거(Clear)하는 특수 역할을 정의하였다.
5.  **Task Achievement (작업 달성 최적화):** Taskboard의 도입에 맞춰 Deliverer라는 새로운 역할을 추가하고, 팀을 여러 그룹으로 나누어 병렬적으로 작업을 수행하는 구조를 설계하였다.

## 📎 Related Works

본 연구는 이전 대회(14회)에서 1위를 차지했던 LFC(Liverpool Formidable Constructors) 팀의 접근 방식을 계승한다. 기존 시스템은 JaCaMo 프레임워크를 기반으로 하며, 에이전트의 행동 결정에는 BDI(Belief-Desire-Intention) 모델과 Fast Downward(FD)라는 자동 계획(Automated Planning) 시스템을 사용하였다.

기존 접근 방식은 PDDL(Planning Domain Definition Language)을 통해 도메인과 문제를 정의하고 최적의 경로를 생성하는 방식이었으나, 이는 계산 비용이 매우 높다는 한계가 있었다. 에이전트 10명 규모에서는 문제가 되지 않았으나, 이번 대회처럼 최대 50명까지 늘어나는 환경에서는 각 에이전트가 매 스텝 플래너를 호출할 경우 제한 시간(4초) 내에 연산을 마칠 수 없는 심각한 성능 저하가 발생한다. 이에 본 논문은 기존의 PDDL 기반 계획 수립 방식을 유지하면서도 이를 효율적으로 가속화하는 캐싱 전략을 통해 차별점을 두었다.

## 🛠️ Methodology

### 전체 시스템 구조
시스템은 JaCaMo 프레임워크의 세 가지 구성 요소인 Jason(BDI 기반 에이전트 언어), CArtAgO(환경 모델링), Moise(조직 모델링)를 통합하여 구축되었다. 에이전트들은 역할(Role) 기반으로 행동하며, 팀 전체의 정보는 CArtAgO의 Team Artifact(블랙보드 형태)를 통해 공유된다.

### 주요 구성 요소 및 전략

#### 1. Cartography (맵 크기 측정)
원형 맵에서는 경계가 없으므로 에이전트들이 맵의 크기를 알 수 없다. 이를 위해 두 명의 에이전트가 한 쌍이 되어 각각 서로 반대 방향(예: 동쪽과 서쪽)으로 이동하며 걸음 수를 센다. 두 에이전트가 다시 만나는 지점에서 각각 이동한 거리의 합을 계산하면 해당 차원의 맵 크기를 구할 수 있다.
$$\text{Map Size} = \text{Initial Distance} + \text{Steps of Agent 1} + \text{Steps of Agent 2}$$
이 정보는 팀 전체에 브로드캐스트되어 이후 모든 에이전트가 맵 좌표를 정규화하는 데 사용된다.

#### 2. Plan Cache (계획 캐시)
계산 비용이 높은 FD Planner의 호출 횟수를 줄이기 위해 $\text{Problem} \rightarrow \text{Plan}$ 형태의 매핑을 저장한다.
- **인코딩 방법:** 에이전트 주변의 $7 \times 7$ (총 61셀) 관측 영역을 1차원 배열로 펼친 후, 각 셀의 상태를 숫자로 치환하여 고유한 문자열 키를 생성한다.
    - 빈 공간/에이전트 위치: 0
    - 장애물: 1
    - 블록: 2
    - 목표 지점: 3
- **절차:** 플래너를 호출하기 전, 현재 상태의 인코딩 키가 캐시 파일로 존재하는지 확인한다. 존재하면 즉시 계획을 불러오고, 없으면(Cache Miss) 플래너를 통해 생성한 뒤 파일로 저장한다.

#### 3. Bullies (공격 전략)
상대 팀을 방해하는 Bully 에이전트는 두 가지 유형으로 나뉜다.
- **Bouncer Bullies:** 경기 초반, 우리 팀이 맵 크기를 측정하는 동안 상대 팀이 먼저 작업을 수행하지 못하도록 특정 목표 지점(Goal Position)을 점유하고 순찰하며 상대의 블록을 제거한다.
- **Hunter Bullies:** 초기 단계 이후, 여러 목표 지점을 이동하며 상대 팀의 활동이 활발한 곳을 찾아내어 방해한다.

#### 4. Task Achievement (작업 수행)
에이전트들은 다음과 같은 역할 분담을 통해 작업을 수행한다.
- **Origin Agent:** 목표 지점의 최하단 셀에 위치하여 구조물의 기초를 잡는다.
- **Retriever:** 디스펜서에서 블록을 가져와 Origin에게 전달한다.
- **Deliverer:** Taskboard에서 작업을 수락하고, 조립이 완료되면 Origin과 위치를 교체하여 최종적으로 작업을 제출한다.
팀 규모에 따라 그룹 수를 $\text{GroupSize} = \text{RoundSize} / 15$로 설정하여 병렬적으로 작업을 수행한다.

## 📊 Results

### 실험 설정
- **데이터셋 및 지표:** MAPC 시나리오의 각 라운드(에이전트 15, 30, 50명)에서의 총 획득 통화량으로 성과를 측정하였다.
- **비교 대상:** 총 5개 팀이 경쟁하였다.

### 결과 분석
MLFC 팀은 최종적으로 전체 **2위**를 기록하였다. 하지만 경기 일차별로 극명한 성능 차이가 나타났다.

- **Day 1 (저조한 성과):** 6라운드 중 1승 1무 4패를 기록하였다. 특히 에이전트 수가 30, 50명으로 늘어난 라운드에서 치명적인 성능 저하가 발생했다. 이는 캐시 데이터가 부족한 상태에서 너무 많은 에이전트가 동시에 FD Planner를 호출함으로써 CPU 과부하가 발생했고, 이로 인해 4초라는 제한 시간을 초과하여 에이전트들이 서버와의 동기화를 잃고 중단되었기 때문이다.
- **Day 2 (우수한 성과):** 6라운드 모두 승리하였다. 코드 수정 없이도 성능이 향상된 이유는 Day 1의 경기 경험을 통해 **Plan Cache가 충분히 구축되었기 때문**이다. 대부분의 계획 요청이 캐시에서 즉시 해결됨으로써 계산 부하가 획기적으로 줄어들었고, 에이전트 중단 현상이 사라지며 안정적으로 작업을 완수할 수 있었다.

## 🧠 Insights & Discussion

### 강점 및 성과
본 연구는 복잡한 AI Planning 시스템을 실제 경쟁 환경의 제약 조건(실시간성, 확장성) 내에서 작동시키기 위해 캐싱 전략을 도입함으로써 성능 문제를 해결하였다. 또한 CSP를 이용한 형식 검증을 통해 다수 에이전트 환경에서도 지도 병합 프로토콜이 교착 상태(Deadlock) 없이 정확하게 작동함을 보장하였다.

### 한계 및 비판적 해석
1.  **캐시 의존성:** 시스템의 성능이 코드의 효율성보다는 '누적된 캐시 데이터'에 과도하게 의존하는 경향을 보인다. 이는 새로운 환경이나 파라미터 변경 시 초기 성능이 급격히 떨어질 수 있음을 의미한다.
2.  **결함 허용 능력(Fault Tolerance) 부족:** 에이전트 한 명의 타임아웃이 팀 전체의 재시작으로 이어지는 구조적 취약성이 발견되었다. 개별 에이전트가 독립적으로 복구하거나 재연결될 수 있는 메커니즘이 부재하다.
3.  **테스트 부족:** 15명 규모의 테스트에 치중하여 30, 50명 규모에서의 캐시 히트율(Cache Hit Rate)과 연산 부하를 사전에 충분히 예측하지 못했다.

## 📌 TL;DR

본 논문은 제15회 MAPC 대회에서 원형 맵, 에이전트 수 증가, Taskboard 도입이라는 환경 변화에 대응하기 위한 전략을 제시한다. 특히 **Plan Cache**를 통해 PDDL 기반 자동 계획의 높은 계산 비용 문제를 해결하여 최대 50명의 에이전트를 효율적으로 운용하였으며, **Cartography**와 **Bullies** 전략을 통해 환경 적응력과 경쟁력을 높였다. 결과적으로 캐시가 충분히 구축된 후에는 압도적인 성능을 보이며 최종 2위를 달성하였으나, 향후 개별 에이전트의 결함 허용 능력(Fault Tolerance) 개선이 필요함을 시사한다.
