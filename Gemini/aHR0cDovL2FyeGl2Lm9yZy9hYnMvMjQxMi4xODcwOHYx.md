# CAG: Chunked Augmented Generation for Google Chrome's Built-in Gemini Nano

Vivek Vellaiyappan Surulimuthu, Aditya Karnam Gururaj Rao (2024)

## 🧩 Problem to Solve

본 논문은 Google Chrome 브라우저에 내장된 Gemini Nano 모델이 가진 **제한적인 컨텍스트 윈도우(Context Window)** 문제를 해결하고자 한다. Chrome의 Gemini Nano 통합은 외부 API 의존성을 제거하고 로컬 처리를 통해 개인정보 보호를 강화하며 AI 접근성을 높였다는 점에서 혁신적이다. 그러나 브라우저의 성능 제약과 모델 역량 사이의 균형을 맞추기 위해 설정된 좁은 컨텍스트 윈도우는 방대한 양의 문서나 코드베이스와 같은 대규모 입력을 처리하는 데 큰 걸림돌이 된다.

따라서 본 연구의 목표는 지능적인 입력 청킹(Chunking) 및 처리 전략을 통해 브라우저의 리소스 제약 내에서 모델의 성능을 유지하면서도, 대규모 콘텐츠를 효율적으로 처리할 수 있는 **Chunked Augmented Generation (CAG)** 아키텍처를 제안하는 것이다.

## ✨ Key Contributions

본 논문의 핵심 기여는 브라우저 환경의 특수성을 고려하여 대규모 텍스트를 처리할 수 있도록 설계된 `cag-js` 라이브러리와 그 기반 아키텍처인 CAG를 제안한 것이다. 

중심적인 설계 아이디어는 대규모 입력을 의미적 일관성을 유지하며 작은 단위로 나누는 **지능적 청킹(Intelligent Chunking)**과, 이를 처리하는 두 가지 경로인 **순차적 생성(Sequential Generation)** 및 **재귀적 생성(Recursive Generation)** 전략을 도입한 것이다. 이를 통해 사용자는 외부 API 호출 없이 브라우저 내부에서만 대규모 문서의 요약 및 확장과 같은 고도화된 AI 기능을 수행할 수 있다.

## 📎 Related Works

논문에서는 컨텍스트 윈도우 확장 기술과 브라우저 기반 AI 처리라는 두 가지 관점에서 관련 연구를 살펴본다.

1. **컨텍스트 윈도우 확장 기술**: YARN과 같은 동적 토큰 선택 방식이나 다양한 메모리 메커니즘을 통한 확장 기법들이 존재한다. 하지만 이러한 연구들은 주로 서버 사이드 모델을 대상으로 하며, 브라우저와 같은 극심한 리소스 제약 환경을 충분히 고려하지 않았다.
2. **브라우저 기반 AI 처리**: WebAssembly(Wasm)와 WebGPU를 활용한 추론 최적화 연구들이 진행되었으며, Chrome의 Gemini Nano 통합이 그 정점에 있다. 그러나 여전히 하드웨어 가속의 가변성과 메모리 관리의 어려움이라는 한계가 존재한다.

CAG는 기존의 서버 중심 확장 기법과 달리 브라우저의 메모리 제약에 최적화된 **점진적 처리(Progressive Processing)**와 **로컬 프라이버시 보존**에 집중함으로써 기존 접근 방식과 차별화된다.

## 🛠️ Methodology

CAG 아키텍처는 `cag-js` 라이브러리를 통해 구현되었으며, 크게 텍스트 청킹 시스템, 처리 파이프라인, 출력 관리 시스템의 세 가지 구성 요소로 이루어진다.

### 1. 전체 파이프라인 및 구성 요소
- **Text Chunking System**: LangChain의 `RecursiveCharacterTextSplitter`를 사용하여 대규모 텍스트를 관리 가능한 크기의 청크(Chunk)로 분할한다. 이때 청크 크기(Chunk size)와 오버랩(Overlap) 파라미터를 조절하여 청크 간의 경계에서 의미적 단절이 일어나지 않도록 유지한다.
- **Processing Pipeline**: 분할된 청크를 Gemini Nano 모델에 전달하여 처리한다.
- **Output Management System**: 처리된 청크들을 결합하며, 설정된 길이 및 품질 임계값에 따라 추가적인 정제(Refinement) 반복 여부를 결정한다.

### 2. 처리 전략 (Processing Strategies)
CAG는 목적에 따라 두 가지 생성 경로를 제공한다.

- **순차적 생성 (Sequential Generation)**: 각 청크를 선형적으로 처리하고 독립적으로 결과를 생성한 뒤 최종적으로 결합하는 방식이다. 청크 간의 독립성이 허용되는 단순 작업에 적합하다.
- **재귀적 생성 (Recursive Generation)**: 생성된 결과물이 목표 토큰 제한이나 반복 횟수 제한에 도달할 때까지 여러 번의 정제 과정을 거치는 반복적 모델이다. 대규모 컨텍스트 전반에 걸쳐 일관성을 유지해야 하는 복잡한 작업에 사용된다.

### 3. 주요 메커니즘 및 최적화
- **Prompt Template System**: 모든 청크에 일관된 지침을 전달하면서도, 각 세그먼트가 전체 생성 목표와 정렬될 수 있도록 동적인 프롬프트 템플릿을 사용한다.
- **Resource Management**: 모델 인스턴스를 처리 직전에 초기화하고 처리 후 즉시 파괴함으로써 브라우저의 메모리 누수를 방지하고 안정성을 확보한다.

## 📊 Results

### 1. 실험 설정 및 측정 지표
본 연구에서는 Wikipedia 기사를 활용하여 입력 크기에 따른 성능을 벤치마킹하였다. 특히 모델의 컨텍스트 윈도우 크기를 기준으로 콘텐츠의 상대적 길이를 측정하는 **Context Window Length Quotient (CWQ)** 지표를 정의하였다.

$$CWQ = \frac{L}{T \times C}$$

여기서 $L$은 콘텐츠의 문자 길이, $T$는 기본 토큰 윈도우 크기(6,144 tokens), $C$는 문자-토큰 비율(약 4 characters/token)을 의미한다. 이에 따라 데이터를 5가지 범주(Small, Medium, Large, ExtraLarge, Humongous)로 분류하였다.

### 2. 정량적 결과
- **압축 효율 (Compression Ratio)**: 입력 텍스트를 요약할 때의 압축률을 측정하였다. 규모가 커질수록 압축률이 높아지는 경향을 보였으며, 특히 **Humongous(CWQ > 4)** 범주에서는 최대 $97\%$의 압축률을 기록하였다.
- **의미적 보존 (ROUGE Scores)**: ROUGE-N, ROUGE-L, ROUGE-S 지표를 통해 요약 결과의 품질을 평가하였다.
    - **Small/Medium**: 높은 ROUGE-N 점수를 보이며 매우 안정적인 성능을 보였다.
    - **Large/ExtraLarge**: ROUGE-N 점수가 $0.4 \sim 0.6$ 사이에서 진동하며, 콘텐츠의 복잡도에 따라 보존 능력이 달라지는 양상을 보였다.
    - **Humongous**: 의미적 일관성이 다소 저하되는 경향을 보였으나, 여전히 수용 가능한 수준의 ROUGE-L 점수를 유지하였다.
- **시스템 성능**: 
    - **지연 시간 (Latency)**: Small(2초 미만) $\rightarrow$ Medium(3.8초) $\rightarrow$ Large(5.2초) $\rightarrow$ ExtraLarge(7.8초) $\rightarrow$ Humongous(12.4초)로 입력 크기에 따라 선형적으로 증가하였다.
    - **메모리 사용량**: 가장 큰 Humongous 데이터 처리 시에도 브라우저 가용 리소스의 $45\%$ 미만을 사용하여 효율적인 메모리 관리를 입증하였다.

## 🧠 Insights & Discussion

### 강점 및 성과
CAG는 브라우저라는 극도로 제한된 환경에서 Gemini Nano의 컨텍스트 윈도우 한계를 효과적으로 극복하였다. 특히 기술 문서와 같이 구조화된 텍스트에서 매우 높은 압축 효율과 의미 보존 능력을 보여주었다. 외부 서버 없이 로컬에서 대규모 데이터를 처리함으로써 프라이버시와 비용 문제를 동시에 해결한 점이 고무적이다.

### 한계 및 비판적 해석
실험 결과에서 나타나듯이, 입력 크기가 'Humongous' 수준으로 커질수록 ROUGE 점수가 하락하며 의미적 일관성이 떨어지는 현상이 관찰된다. 이는 단순한 청킹-결합 구조가 매우 긴 문서의 전역적 문맥(Global Context)을 완벽하게 캡처하기에는 한계가 있음을 시사한다. 또한, 청크 간의 조율(Reconciliation) 오버헤드가 전체 처리 시간의 최대 $25\%$까지 차지한다는 점은 향후 최적화가 필요한 부분이다.

### 결론 및 논의
본 연구는 로컬 LLM의 컨텍스트 제한을 소프트웨어 아키텍처 수준에서 해결할 수 있음을 보여주었다. 이는 향후 웹 브라우저가 단순한 뷰어가 아닌, 강력한 로컬 AI 워크스테이션으로 진화하는 데 중요한 기초가 될 것이다.

## 📌 TL;DR

본 논문은 Chrome 내장 Gemini Nano의 좁은 컨텍스트 윈도우 문제를 해결하기 위해, 텍스트를 지능적으로 분할하고 순차적/재귀적으로 처리하는 **CAG(Chunked Augmented Generation)** 아키텍처를 제안한다. Wikipedia 데이터를 이용한 실험 결과, 매우 큰 문서(CWQ > 4)에 대해서도 최대 $97\%$의 압축률과 안정적인 메모리 효율을 보였으며, 이를 통해 외부 API 없이 브라우저 내에서 프라이버시를 보장하며 대규모 문서를 요약 및 확장할 수 있음을 입증하였다. 이 연구는 리소스가 제한된 엣지 디바이스에서의 LLM 활용 방안에 중요한 시사점을 제공한다.
