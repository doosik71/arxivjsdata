# The Multi-Agent Programming Contest: A résumé Comparing agent systems 2005-2019

Tobias Ahlbrecht, Jürgen Dix, Niklas Fiekas, and Tabajara Krausburg (2020)

## 🧩 Problem to Solve

본 논문은 2005년부터 2019년까지 진행된 Multi-Agent Programming Contest(이하 MAPC)의 전반적인 과정과 그 성과를 분석한다. 연구의 핵심 문제는 분산된 환경에서 자율적으로 동작하는 지능형 에이전트 시스템의 잠재력을 조사하고, 특히 에이전트 지향 프로그래밍 언어(Agent-oriented programming languages)가 전통적인 클래식 프로그래밍 언어에 비해 어떤 이점이 있는지, 그리고 어떤 문제 상황에서 그 효율성이 극대화되는지를 정량적, 정성적으로 비교 평가하는 것이다.

단순히 특정 시나리오의 최적 솔루션을 찾는 것이 아니라, 복잡한 과제를 해결하기 위해 필요한 시스템의 기능과 기술적 역량을 평가하는 데 목적이 있다. 즉, 에이전트 기반의 자율성, 통신, 협력 및 반응성(reactiveness)과 주도성(proactiveness)의 균형이 실제 구현 단계에서 얼마나 유용한지를 검증하고자 한다.

## ✨ Key Contributions

본 연구의 주요 기여는 15년에 걸친 MAPC의 운영 경험을 통해 에이전트 시스템의 성능을 평가할 수 있는 벤치마크 시나리오들을 설계하고, 이를 실행하기 위한 시뮬레이션 프레임워크인 MASSim을 구축 및 발전시킨 점에 있다.

특히, 에이전트들이 단순한 개별 동작을 넘어 상호 협력해야만 해결 가능한 시나리오(예: Cows and Cowboys, Agents Assemble)를 지속적으로 설계함으로써, 참가자들이 에이전트 지향 프로그래밍의 특성을 실제로 활용하도록 유도하였다. 또한, 다양한 에이전트 플랫폼(GOAL, JaCaMo, Jason 등)과 일반 Java 구현체를 동일한 환경에서 비교함으로써, 전용 에이전트 플랫폼의 성숙도와 사용성을 평가할 수 있는 실증적인 데이터를 제공하였다.

## 📎 Related Works

논문에서는 MAS(Multi-Agent Systems)와 관련된 다양한 경쟁 플랫폼을 언급하며 MAPC와의 차별점을 설명한다.

- **(Power) Trading Agent Competition**: 에너지 시장의 거래 시나리오를 다루지만, 각 팀이 단일 브로커 에이전트만 사용하므로 협력이나 조정(coordination) 요소가 부족하다.
- **RoboCup Simulation Leagues**: 축구, 재난 구조 등 다양한 도메인을 다루며 매우 유명한 대회이다. 하지만 일부 리그(예: RoboCup Rescue)는 추상적인 에이전트보다는 가상 로봇의 물리적 특성이나 노이즈 섞인 센서 등 리얼리즘에 치중하는 경향이 있다.
- **International Planning Competition (IPC)**: 계획(Planning) 능력에 집중하는 반면, MAPC에서 계획은 에이전트 시스템의 여러 구성 요소 중 하나일 뿐이다.
- **General Game Playing (GGP)**: 특정 도메인이 아닌, 임의의 게임 규칙을 받았을 때 이를 해결하는 일반 AI의 능력에 초점을 맞춘다.
- **Commercial Game AI (Mario AI, StarCraft AI 등)**: 주로 게임 AI 알고리즘의 벤치마크를 목적으로 한다.
- **Intention Progression Competition (IPC)**: 에이전트의 의도(Intention)와 계획을 어떻게 진행할 것인가라는 구체적인 문제에 집중하며, 이는 MAPC 솔루션의 하위 문제로 볼 수 있다.

MAPC는 이러한 기존 대회들과 달리, 특정 알고리즘의 효율성보다는 에이전트 지향 언어와 플랫폼의 범용성과 적합성을 비교하는 데 주안점을 둔다.

## 🛠️ Methodology

### MASSim 프레임워크 구조

MASSim은 Java로 구현된 확장 가능한 시뮬레이션 서버로, 다음과 같은 작동 메커니즘을 가진다.

1. **실행 루프**: 시뮬레이션은 단계별(step-by-step)로 진행된다.
2. **Percepts 전달**: 서버는 각 에이전트에게 현재 환경 상태를 Percepts 형태로 전송한다.
3. **Action 수집**: 에이전트는 이에 대응하는 Action을 결정하여 서버로 전송한다.
4. **상태 업데이트**: 모든 에이전트의 Action이 수집되거나 타임아웃(4초)이 발생하면, 서버는 이를 동시에 실행하고 다음 단계의 상태를 계산한다.
5. **통신**: 서버와 에이전트 간의 데이터 교환은 JSON 형식을 사용하며, 원격 연결 방식을 통해 참가자가 자신의 인프라에서 에이전트를 실행하도록 한다.

### 2019년 시나리오: Agents Assemble

최신 시나리오인 'Agents Assemble'의 상세 내용은 다음과 같다.

- **환경**: 장애물이 존재하는 그리드(Grid) 구조이다.
- **목표**: 에이전트들이 그리드 상의 블록(Blocks)을 찾아 수집하고, 무작위로 부여되는 특정 형태(Shape)로 조립하여 목표 구역(Goal zone)에 전달함으로써 점수를 획득하는 것이다.
- **에이전트 능력**:
  - 4방향 이동 및 90도 회전이 가능하다.
  - 에이전트는 4개의 팔을 가지고 있어 블록을 잡거나 연결할 수 있다.
  - 두 명의 에이전트가 각각 블록을 잡고 있을 때 `connect` 액션을 통해 블록 간의 연결이 가능하다.
  - `clear` 액션을 통해 특정 셀의 장애물이나 블록을 제거하거나 상대 에이전트를 무력화할 수 있다. 단, 이 액션은 여러 번 실행하여 '충전'해야 효과가 나타난다.
- **제약 사항**: 에이전트는 절대 좌표가 아닌 **상대 좌표**만을 인지하며, 시야 범위가 5셀로 제한된다. 또한, 시뮬레이션 도중 무작위로 `clear` 이벤트가 발생하여 환경이 동적으로 변화한다.

## 📊 Results

### 2019년 대회 결과

4개 팀(FIT BUT, GOAL-DTU, LFC, TRG)이 참여하였으며, 각 팀은 서로 3번의 시뮬레이션을 수행하는 총 9번의 경기를 치렀다.

- **최종 순위**: LFC(JaCaMo 사용)가 22점으로 우승하였으며, 뒤이어 FIT BUT(Java), GOAL-DTU(GOAL), TRG(Jason) 순으로 기록되었다.
- **정량적 결과**: LFC는 가장 많은 총점(1790점)을 기록하며 압도적인 성능을 보였다.

### 팀별 전략 분석

- **LFC (우승)**: 한 명의 에이전트가 목표 구역에서 대기하며, 다른 에이전트들이 가져온 블록들을 그곳에서 최종 조립하는 전략을 사용하였다. 또한 `clear` 액션을 이용해 장애물을 뚫고 직선 경로를 만드는 효율적인 기법을 구현하였으며, 탐색자(Explorer)에서 조립자(Assembler)로 역할을 변경하는 동적 역할 할당을 적용하였다.
- **FIT BUT**: 이동 중에 에이전트들이 만나 블록을 연결하는 방식을 사용하여 복잡한 형태를 함께 운반하였다.
- **GOAL-DTU**: 한 번에 최대 4개의 블록을 수집하여 운반하는 공격적인 수집 전략을 사용하였으나, 후반부 좁아진 경로에서 이동에 어려움을 겪었다.
- **TRG**: 일부 에이전트가 목표 구역을 방어하며 상대 팀을 무력화하는 하이브리드 전략을 시도하였으나, 상대 팀이 이를 우회하면서 큰 효과를 거두지 못했다.

## 🧠 Insights & Discussion

### 강점 및 성과

본 연구는 15년간의 데이터를 통해 전용 에이전트 플랫폼(JaCaMo, GOAL 등)을 사용한 팀들이 일반적인 ad-hoc Java 구현체보다 우수한 성능을 보이는 경향이 있음을 확인하였다. 이는 에이전트 플랫폼의 성숙도가 높아졌으며, 복잡한 협력 과제를 구현하는 데 있어 플랫폼이 제공하는 추상화 수준이 유리하게 작용했음을 시사한다.

### 한계 및 비판적 해석

1. **협력의 강제성 문제**: 연구자들은 팀들이 항상 '최소 저항 경로(Path of least resistance)'를 선택한다는 점을 발견하였다. 즉, 협력이 가능하더라도 단순한 개별 동작으로 해결 가능하다면 협력 기능을 사용하지 않는다. 따라서 진정한 MAS의 능력을 측정하려면 협력을 '보상'하는 것이 아니라 '강제'하는 시나리오 설계가 필수적이다.
2. **디버깅의 어려움**: 모든 참가 팀이 공통적으로 지적한 문제는 MAS의 디버깅 난이도이다. 20명의 에이전트가 각자 다른 상태를 가지고 동작할 때, 이를 추적하고 수정하는 표준화된 도구가 부족하여 많은 시간이 낭비되었다.
3. **Killer Application의 부재**: 15년의 연구에도 불구하고, 에이전트 기반 접근 방식이 전통적인 방식보다 압도적으로 우월함을 증명할 '킬러 애플리케이션'을 찾지 못했다. 이는 MAS의 장점이 성능(Performance)보다는 재사용성, 유지보수성, 정당성 검증(Model-checking) 등 소프트웨어 공학적 측면에 있을 가능성이 크기 때문이다.

## 📌 TL;DR

본 논문은 2005년부터 2019년까지의 MAPC 운영 사례를 통해 분산 자율 에이전트 시스템의 벤치마킹 결과를 분석한 보고서이다. 전용 에이전트 플랫폼이 일반 구현체보다 복잡한 협력 과제 해결에 유리함을 보였으며, 특히 2019년 대회에서는 동적 역할 할당과 효율적인 경로 확보 전략을 사용한 LFC 팀이 우승하였다. 이 연구는 MAS의 실용적 가치를 입증하기 위해서는 단순한 성능 경쟁을 넘어, 협력을 강제하는 정교한 시나리오 설계와 강력한 디버깅 도구의 개발이 필요함을 시사한다.
