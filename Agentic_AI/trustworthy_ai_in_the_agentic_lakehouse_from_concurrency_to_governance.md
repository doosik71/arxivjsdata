# Trustworthy AI in the Agentic Lakehouse: from Concurrency to Governance

Jacopo Tagliabue, Federico Bianchi, Ciro Greco (2025)

## 🧩 Problem to Solve

본 논문은 기업 환경에서 AI 에이전트가 프로덕션 데이터에 직접 접근하여 작업하는 것을 가로막는 핵심 장애물인 **신뢰성(Trust)**과 **거버넌스(Governance)** 문제를 해결하고자 한다. 현재의 데이터 레이크하우스(Lakehouse)는 소수의 숙련된 인간 엔지니어 팀을 위해 설계되었으며, 수많은 저렴한 AI 에이전트 집단(swarm of agents)이 생성하는 접근 패턴을 처리하기에는 부적합하다.

특히 에이전트가 실수로 테이블을 삭제하거나, 환각(hallucination)으로 인해 오염된 데이터를 레이크에 삽입하는 경우와 같은 치명적인 결과에 대한 우려가 크다. 또한, 기존의 OLAP 시스템은 에이전트 규모의 동시성(concurrency) 제어를 처리하도록 설계되지 않았다. 따라서 본 논문의 목표는 인프라 수준에서 데이터와 계산의 격리(isolation)를 해결함으로써, 에이전트가 안전하게 작동할 수 있는 '에이전트 우선(agent-first)' 레이크하우스 구조를 제안하는 것이다.

## ✨ Key Contributions

본 논문의 핵심 아이디어는 **거버넌스 문제를 인프라 문제로 치환**하여 해결하는 것이다. 즉, 고차원의 정책적 거버넌스에 집중하기보다, 데이터베이스의 **MVCC(Multi-Version Concurrency Control)** 개념을 레이크하우스에 맞게 재설계하여 동시성 제어와 격리를 완벽히 구현하면, 거버넌스는 자연스럽게 API 기반의 접근 제어 문제로 단순화된다는 직관을 제시한다.

이를 위해 저자들은 데이터 격리를 위한 Git 스타일의 브랜칭(branching)과 계산 격리를 위한 FaaS(Function-as-a-Service) 모델을 결합한 **Bauplan**이라는 시스템 설계를 제안한다.

## 📎 Related Works

### 기존 접근 방식 및 한계

1. **전통적인 MVCC 데이터베이스**: Postgres와 같은 모놀리식 SQL 데이터베이스는 스냅샷과 격리된 프로세스를 통해 강력한 동시성 제어를 제공한다. 그러나 이는 스토리지와 계산이 통합된 구조이므로, 스토리지와 계산이 분리된 분산 환경인 레이크하우스에 그대로 적용할 수 없다.
2. **Apache Iceberg 기반 레이크하우스**: 현재 많은 레이크하우스가 사용하는 Iceberg는 테이블 단위의 트랜잭션을 보장한다. 하지만 실제 데이터 파이프라인은 여러 테이블에 걸쳐 작업하므로, 개별 테이블의 트랜잭션만으로는 파이프라인 전체의 일관성을 보장할 수 없다. (예: 노드 1, 2는 성공했으나 노드 3에서 실패한 경우, 데이터 레이크는 불완전한 상태가 됨)
3. **Airflow 등 기존 워크플로우 도구**: AWS S3와 Airflow를 이용한 파이프라인은 사용자가 직접 스토리지 계층과 DAG 프로세스를 관리해야 한다. 이로 인해 트랜잭션 보장이 불가능하며, 물리적 파일에 직접 접근해야 하므로 에이전트에게 과도한 권한을 부여해야 하는 보안 취약점이 발생한다.

## 🛠️ Methodology

저자들은 MVCC의 세 가지 핵심 요소인 데이터 격리, 계산 격리, 프로그래밍 추상화를 레이크하우스 환경에 맞게 재구현한 **Bauplan**을 제안한다.

### 1. 데이터 격리 (Data Isolation)

Bauplan은 모든 쓰기 작업을 불변(immutable)하게 기록하며, Git과 유사한 **Copy-on-Write 브랜칭 메커니즘**을 도입한다.

- **브랜치 및 커밋**: 각 스냅샷은 부모를 가진 커밋으로 식별되며, 브랜치의 HEAD를 통해 최신 상태를 관리한다.
- **원자적 병합(Atomic Merge)**: 단일 테이블 단위의 커밋(commit)과 별개로, 여러 테이블에 걸친 변경 사항을 한 번에 반영하는 병합(merge) 단계를 두어 파이프라인 전체의 원자성을 보장한다.

### 2. 계산 격리 (Compute Isolation)

이질적인 런타임(SQL, Python 3.10, 3.11 등)을 수용하기 위해 **FaaS(Function-as-a-Service)** 모델을 채택한다.

- 각 함수는 컨테이너화된 독립적인 환경에서 실행되며, 외부 인터넷 접속이 완전히 차단되어 패키지 설치 등을 통한 공격 벡터를 최소화한다.
- 하드웨어, OS, 런타임 관리는 플랫폼이 담당하고 에이전트는 오직 코드만 제공한다.

### 3. 프로그래밍 추상화 및 실행 흐름 (Programming Abstractions)

에이전트가 쉽게 인식할 수 있도록 Python 데코레이터를 이용한 선언적 I/O 및 인프라-코드(Infra-as-code) 방식을 사용한다.

- **선언적 I/O**: 함수가 물리적 파일 경로가 아닌 논리적 테이블(Model)을 입력과 출력으로 받도록 설계하여 물리 계층으로의 직접 접근을 차단한다.
- **통합 실행 API**: `bauplan.run(pipeline)` API를 통해 다음과 같은 절차로 실행된다.
    1. 메인 브랜치로부터 자동으로 **임시 브랜치(temporary branch)**를 생성한다.
    2. S3에서 소스 테이블 데이터를 가져와 FaaS 런타임에서 코드를 실행한다.
    3. 실행 결과물을 임시 브랜치에 기록한다.
    4. 모든 작업이 성공하면 임시 브랜치를 메인 브랜치로 **원자적으로 병합(Atomic Merge)**하며, 실패 시 임시 브랜치를 유지하고 메인 브랜치는 전혀 수정하지 않는다.

### 4. 거버넌스 체계

위의 메커니즘을 통해 다음과 같이 거버넌스를 구현한다.

- **데이터 거버넌스**: 선언적 I/O를 통해 SQL-like 권한 확인 계층을 두고, 병합 전 검증 단계를 거쳐 프로덕션 데이터를 보호한다.
- **계산 거버넌스**: 데코레이터에 명시된 패키지 목록을 화이트리스트와 대조하여 위험한 패키지 설치를 원천 차단한다.
- **API 거버넌스**: 모든 작업이 통합 API를 통해 수행되므로, RBAC(Role-Based Access Control)를 통해 정밀한 권한 제어가 가능하다.

## 📊 Results

본 논문은 이론적 제안을 뒷받침하기 위해 **자가 치유 파이프라인(Self-healing pipelines)**이라는 구체적인 구현 사례를 제시한다.

### 실험 시나리오 및 흐름

데이터 파이프라인 실행 중 오류가 발생했을 때, 에이전트가 이를 자동으로 수정하는 프로세스는 다음과 같다.

1. **오류 감지 및 트리거**: 실행 실패 시 에이전트의 ReAct 루프가 시작된다.
2. **샌드박스 해결**: 에이전트는 Bauplan API를 사용하여 해결 코드를 작성하고 실행한다. 이때 모든 쓰기 작업은 별도의 데이터 브랜치에서 이루어지므로 프로덕션 데이터는 안전하게 보호된다.
3. **검증 및 확인**:
    - **자동 검증**: 플랫폼에 등록된 'Verifier(수락 기준)'가 에이전트의 수정 결과가 최소 기준을 충족하는지 독립적으로 확인한다.
    - **인간 확인(Human-in-the-loop)**: 검증을 통과한 브랜치를 인간 엔지니어가 최종 리뷰한다.
4. **최종 반영**: 리뷰가 승인되면 Copy-on-Write 병합을 통해 수정 사항이 메인 브랜치에 효율적으로 반영된다.

이 과정은 에이전트의 지능에만 의존하는 것이 아니라, 인프라 수준에서 제공되는 브랜치-병합-격리 구조 덕분에 안전하게 수행될 수 있음을 보여준다.

## 🧠 Insights & Discussion

### 강점 및 통찰

- **인프라 기반의 신뢰 구축**: 신뢰성 문제를 LLM의 성능 향상(지능)으로 해결하려는 일반적인 접근과 달리, 데이터 시스템의 고전적인 MVCC 개념을 도입하여 인프라 수준에서 해결책을 제시했다는 점이 매우 탁월하다.
- **공격 표면의 최소화**: 물리적 스토리지 접근을 차단하고 좁은 API 접점만을 제공함으로써, 거버넌스 비용을 획기적으로 낮췄다.

### 한계 및 논의사항

- **성능 오버헤드**: Copy-on-Write 브랜칭과 FaaS 기반의 격리 구조가 대규모 데이터 처리 시 어떤 성능 저하를 가져오는지에 대한 정량적 분석이 부족하다.
- **에이전트의 논리적 오류**: 인프라가 물리적 파괴(테이블 삭제 등)는 막아줄 수 있지만, 에이전트가 논리적으로 잘못된(그러나 문법적으로는 옳은) 데이터를 생성하여 병합을 요청했을 때, 이를 걸러낼 'Verifier'를 누가 어떻게 작성하느냐가 새로운 병목 지점이 될 수 있다.

## 📌 TL;DR

본 논문은 AI 에이전트가 프로덕션 데이터 레이크하우스에서 안전하게 작업하기 위해 필요한 **'에이전트 우선 레이크하우스(Agentic Lakehouse)'**의 설계 원칙을 제안한다. 핵심은 MVCC 개념을 확장하여 **Git 스타일의 데이터 브랜칭**과 **FaaS 기반의 계산 격리**를 구현한 **Bauplan** 시스템이다. 이를 통해 에이전트의 모든 작업은 격리된 환경에서 수행되며, 검증된 결과만 원자적으로 병합되도록 하여, 인프라 수준에서 데이터 거버넌스와 신뢰성을 확보할 수 있음을 보여준다. 이 연구는 향후 자율형 데이터 엔지니어링 시스템의 표준 인프라 구조를 제시하는 데 중요한 역할을 할 것으로 보인다.
