# AutoML-Agent: A Multi-Agent LLM Framework for Full-Pipeline AutoML

Patara Trirat, Wonyong Jeong, Sung Ju Hwang (2025)

## 🧩 Problem to Solve

본 논문은 데이터 수집부터 모델 배포에 이르는 전체 머신러닝(ML) 파이프라인을 자동화하는 AutoML(Automated Machine Learning)의 진입 장벽을 낮추는 것을 목표로 한다. 기존의 AutoML 시스템은 복잡한 도구 설정과 구성 과정에서 상당한 수준의 프로그래밍 전문 지식을 요구하며, 이는 도메인 전문가와 같은 비전문가 사용자에게 큰 진입 장벽이 된다.

최근 대규모 언어 모델(LLM)을 활용하여 자연어 인터페이스를 제공함으로써 이러한 부담을 줄이려는 시도가 있었으나, 기존의 LLM 기반 AutoML 프레임워크들은 다음과 같은 한계를 가진다. 첫째, 특정 파이프라인 단계(예: 특성 공학, 하이퍼파라미터 최적화)나 특정 도메인(예: NLP, CV)에만 국한되어 설계되었다. 둘째, 전체 파이프라인을 고려하지 않고 개별 단계를 독립적으로 처리함으로써 최적의 솔루션을 찾지 못하는 하위 최적(suboptimal) 문제에 직면한다. 셋째, 탐색 과정에서 실제 모델 학습을 반복적으로 수행하여 계산 비용이 매우 높고 속도가 느리다.

따라서 본 연구의 목표는 데이터 검색, 전처리, 모델 선택, 하이퍼파라미터 최적화(HPO), 그리고 배포에 이르는 **전체 파이프라인(Full-Pipeline)을 태스크에 구애받지 않고(Task-Agnostic) 자동화**하는 효율적인 멀티 에이전트 프레임워크를 구축하는 것이다.

## ✨ Key Contributions

본 논문의 핵심 아이디어는 전문화된 LLM 에이전트들의 협업 체계를 구축하고, 실제 학습 없이도 최적의 계획을 탐색하는 전략을 도입하는 것이다. 주요 기여 사항은 다음과 같다.

1. **전체 파이프라인 자동화 프레임워크**: 데이터 검색부터 모델 배포까지의 전 과정을 자동화하는 최초의 태스크 불가지론적(task-agnostic) LLM 기반 AutoML 프레임워크인 `AutoML-Agent`를 제안한다.
2. **Retrieval-Augmented Planning (RAP)**: 외부 API(arXiv, Kaggle 등)를 통해 최신 지식을 검색하고 이를 기반으로 복수의 실행 계획을 수립하는 전략을 도입하여 탐색 범위를 넓히고 최적의 계획을 찾을 가능성을 높였다.
3. **Role-Specific Plan Decomposition 및 Prompting-Based Execution**: 복잡한 전체 계획을 각 에이전트의 역할에 맞는 하위 작업으로 분해하고, 실제 코드를 실행하는 대신 LLM의 In-context learning 능력을 활용한 '가상 실행(Pseudo-execution)'을 통해 검색 효율성을 극대화하였다.
4. **다단계 검증(Multi-stage Verification)**: 요청 검증(Request), 실행 검증(Execution), 구현 검증(Implementation)의 3단계 필터를 통해 생성된 코드의 정확성을 보장하고, 실패 시 피드백을 통한 계획 수정 과정을 거치게 한다.

## 📎 Related Works

기존의 AutoML 연구는 크게 전통적인 프로그래밍 기반 방식과 최근의 LLM 기반 방식으로 나뉜다. AutoGluon과 같은 전통적 AutoML은 강력한 성능을 보이지만, 설정 과정이 복잡하여 전문가의 개입이 필수적이다.

최근 제안된 LLM 기반 프레임워크들(예: DS-Agent, Agent K 등)은 자연어 인터페이스를 통해 사용성을 높였으나, 다음과 같은 차별점이 존재한다.

- **범위의 제한**: 대부분의 기존 연구는 하이퍼파라미터 최적화(HPO)나 특성 공학 등 특정 단계에만 집중하거나, 특정 데이터 모달리티에 국한되어 있다.
- **계산 비용**: SELA와 같이 Monte Carlo Tree Search(MCTS)를 사용하는 방식은 반복적인 모델 학습을 통해 성능을 개선하지만, 시간과 비용 소모가 극심하다.

`AutoML-Agent`는 이러한 한계를 극복하기 위해 전 파이프라인을 통합 관리하며, 특히 **Training-Free Search** 방식을 통해 실제 학습 횟수를 최소화하면서도 외부 지식 검색을 통해 고성능 모델을 빠르게 찾아낸다는 점에서 기존 연구와 차별화된다.

## 🛠️ Methodology

### 1. 시스템 구조 및 에이전트 구성

`AutoML-Agent`는 `Agent Manager`를 중심으로 5개의 전문 에이전트가 협력하는 구조이다.

- **Agent Manager ($A_{mgr}$)**: 전체 프로세스를 조율하는 컨트롤 타워이다. 사용자와 상호작용하고, RAP를 통해 글로벌 계획을 수립하며, 각 에이전트에게 작업을 배분하고 결과를 검증한다.
- **Prompt Agent ($A_p$)**: 사용자 지시문을 분석하여 표준화된 JSON 객체로 파싱한다. 이는 이후 단계에서 모든 에이전트가 공유하는 컨텍스트가 된다.
- **Data Agent ($A_d$)**: 데이터 검색, 전처리, 증강 및 데이터 특성 분석을 담당한다.
- **Model Agent ($A_m$)**: 모델 검색, HPO, 모델 프로파일링 및 후보군 랭킹을 담당한다.
- **Operation Agent ($A_o$)**: 검증된 최적 솔루션을 바탕으로 실제 실행 가능한 파이썬 코드를 생성하고 배포 엔드포인트를 구축한다.

### 2. 전체 파이프라인 절차

시스템은 다음의 3단계 과정을 거쳐 작동한다.

**가. 초기화 및 계획 단계 (Initialization & Planning)**
사용자의 요청이 들어오면 `Request Verification`을 통해 유효성을 검토한다. 이후 $A_p$가 요청을 JSON으로 파싱하면, $A_{mgr}$이 **Retrieval-Augmented Planning (RAP)**을 수행한다. RAP는 외부 API를 통해 관련 논문이나 코드를 검색하고, 이를 바탕으로 서로 다른 시나리오를 가진 $P$개의 독립적인 실행 계획 $\mathcal{P} = \{p_1, \dots, p_P\}$를 수립한다.

**나. 실행 단계 (Execution)**
수립된 계획은 **Plan Decomposition (PD)** 과정을 통해 각 에이전트의 역할에 맞는 하위 작업으로 분해된다.

- **Pseudo Data Analysis**: $A_d$는 실제 코드를 실행하지 않고, 데이터셋의 특성과 사용자 요구사항을 고려하여 전처리 및 증강 과정을 '설명'하고 예상 결과 $O^d_i$를 도출한다.
- **Training-Free Model Search & HPO**: $A_m$은 $A_d$의 결과와 외부 지식을 바탕으로 최적의 모델 아키텍처와 하이퍼파라미터 수치를 추론하여 top-k개의 후보 모델 $O^m_i$를 선정한다.

**다. 구현 및 배포 단계 (Implementation & Deployment)**
$A_{mgr}$이 가장 유망한 결과 $O^*$를 선택하여 $A_o$에게 전달하면, $A_o$는 실제 작동하는 전체 파이프라인 코드를 작성한다. 이 코드는 마지막으로 `Implementation Verification`을 거쳐 배포 가능한 상태인지 확인된다.

### 3. 주요 방정식 및 지표

본 논문은 모델의 성공 여부와 성능을 동시에 평가하기 위해 **Comprehensive Score (CS)**를 사용한다.

$$CS = 0.5 \times SR + 0.5 \times NPS$$

여기서 $SR$ (Success Rate)은 코드가 오류 없이 실행되고 사용자 제약 조건을 만족했는지를 나타내는 점수이며, $NPS$ (Normalized Performance Score)는 모델의 실제 성능 지표이다. 특히 손실 기반 지표(예: RMSLE)의 경우, 다음과 같이 정규화하여 계산한다.

$$NPS = \frac{1}{1+s}$$

## 📊 Results

### 1. 실험 설정

- **데이터셋**: 이미지, 텍스트, 정형 데이터, 시계열, 그래프 등 5가지 모달리티의 14개 데이터셋을 사용하였다.
- **비교 대상**: 전문가가 설계한 Human Models, SOTA AutoML인 AutoGluon, 일반 LLM(GPT-3.5, GPT-4), 그리고 LLM 기반 에이전트인 DS-Agent 및 SELA와 비교하였다.
- **평가 설정**: 제약 조건이 없는 'Constraint-Free' 설정과 특정 정확도, 지연 시간, 모델 크기 등의 제약이 있는 'Constraint-Aware' 설정 두 가지로 나누어 평가하였다.

### 2. 주요 결과

- **성공률 (SR)**: `AutoML-Agent`는 특히 제약 조건이 있는 설정에서 평균 87.1%의 SR을 기록하며 모든 베이스라인을 압도하였다. 이는 RAP를 통해 제약 조건을 충족할 수 있는 구체적인 전략을 사전에 수립했기 때문으로 분석된다.
- **다운스트림 성능 (NPS)**: 성공적으로 구축된 모델들의 성능을 비교했을 때, `AutoML-Agent`는 많은 태스크에서 Human Models 및 AutoGluon과 대등하거나 더 높은 성능을 보였다.
- **종합 점수 (CS)**: SR과 NPS를 결합한 CS 지표에서도 모든 데이터셋에 걸쳐 가장 높은 성능을 기록하였다.

### 3. 추가 분석

- **효율성**: SELA(MCTS 기반)와 비교했을 때, 실제 학습 과정을 생략한 `AutoML-Agent`는 **약 8배 빠른 탐색 속도**를 보였으며, 성능 면에서도 대등하거나 우수하였다.
- **비용**: 단일 모델 검색 및 배포에 소요되는 평균 시간은 약 525초, 비용은 약 0.30 USD (GPT-4o 기준)로 나타났다.
- **강건성**: 외부 지식에 노이즈를 섞어 주입하는 실험에서도 다단계 검증 메커니즘 덕분에 성능 저하가 거의 없음을 확인하여 시스템의 강건성을 입증하였다.

## 🧠 Insights & Discussion

본 논문은 LLM이 단순한 코드 생성기를 넘어, 전문적인 ML 엔지니어의 추론 과정을 모사하여 전체 파이프라인을 설계할 수 있음을 보여주었다. 특히 **"실제 학습 없이 LLM의 지식만으로 하이퍼파라미터를 추론하고 검증하는 전략"**이 계산 효율성을 획기적으로 높일 수 있음을 증명한 점이 매우 인상적이다.

**강점**으로는 멀티 에이전트 구조를 통해 복잡한 문제를 분해하여 처리함으로써 LLM의 환각(hallucination) 현상을 줄이고, 외부 지식 검색(RAP)을 통해 최신 SOTA 모델을 즉각적으로 반영할 수 있다는 점이 꼽힌다.

**한계점 및 논의사항**으로는 다음과 같은 점들이 언급된다.

- **백본 모델 의존성**: GPT-4와 GPT-3.5 간의 성능 차이가 크게 나타나, 고성능 LLM에 대한 의존도가 높다.
- **스켈레톤 코드의 필요성**: 완전히 새로운 태스크의 경우 기초가 되는 스켈레톤 코드가 없으면 코드 생성 시 환각이 발생할 가능성이 있다.
- **도메인 확장성**: 강화학습(RL)이나 추천 시스템과 같이 일반적인 지도/비지도 학습과 파이프라인이 크게 다른 영역으로 확장하기 위해서는 추가적인 전문 에이전트 설계가 필요하다.

## 📌 TL;DR

`AutoML-Agent`는 데이터 수집부터 모델 배포까지의 전 과정을 자동화하는 멀티 에이전트 LLM 프레임워크이다. **외부 지식 기반의 계획 수립(RAP)**과 **학습 없는 가상 탐색(Training-Free Search)**, 그리고 **3단계 검증 체계**를 통해 계산 비용을 획기적으로 줄이면서도 전문가 수준의 ML 파이프라인을 구축한다. 이 연구는 비전문가가 자연어만으로 고성능 ML 솔루션을 구축할 수 있는 길을 열었으며, 향후 다양한 특수 도메인으로의 AutoML 확장 가능성을 제시한다.
