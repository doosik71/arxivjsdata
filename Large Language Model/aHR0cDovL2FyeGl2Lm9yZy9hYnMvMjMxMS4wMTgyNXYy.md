# Large Language Models to the Rescue: Reducing the Complexity in Scientific Workflow Development Using ChatGPT

Mario Sänger, Ninon De Mecquenem, Katarzyna Ewa Lewińska, Vasilis Bountris, Fabian Lehmann, Ulf Leser, Thomas Kosch (2023)

## 🧩 Problem to Solve

현대 과학 연구에서 대규모 데이터 분석 파이프라인, 즉 Scientific Workflow의 중요성이 증대되고 있다. 이러한 워크플로우 시스템은 대규모 컴퓨팅 클러스터에서의 자동 병렬화를 통해 재현성, 신뢰성 및 확장성을 제공한다. 그러나 실제 구현 과정에서 다음과 같은 심각한 문제들이 발생한다.

첫째, 워크플로우는 수많은 블랙박스 도구(Black-box tools)와 복잡한 인프라 스택(Infrastructure stack)의 결합으로 이루어져 있어 구현 난이도가 매우 높다. 둘째, 워크플로우를 개발하는 주체는 주로 특정 도메인의 전문가(과학자)들이며, 이들은 도메인 지식은 풍부하지만 소프트웨어 개발이나 분산 컴퓨팅에 대한 전문 지식이 부족한 경우가 많다. 셋째, 일반적인 프로그래밍 언어에 비해 공개된 워크플로우 예제가 매우 적어 참고할 수 있는 자료가 부족하다.

따라서 본 논문의 목표는 대규모 언어 모델(LLM), 특히 ChatGPT가 과학적 워크플로우의 이해, 수정 및 확장을 지원함으로써 도메인 전문가들이 겪는 인지적 부하와 구현 복잡성을 줄일 수 있는지 그 효율성을 평가하는 것이다.

## ✨ Key Contributions

본 연구의 핵심 기여는 일반 목적의 LLM이 과학적 워크플로우 개발의 서로 다른 단계(이해 $\rightarrow$ 수정 $\rightarrow$ 확장)에서 어느 정도의 성능을 보이는지 실증적으로 분석했다는 점이다. 연구진은 생물정보학(Bioinformatics)과 지구 관측(Earth Observation)이라는 두 가지 서로 다른 과학 분야에서 실제 워크플로우 스크립트를 활용하여 세 가지 사용자 연구(User Study)를 수행하였다. 이를 통해 LLM이 단순한 설명 및 도구 탐색에는 강점이 있으나, 실제 실행 가능한 코드를 생성하거나 복잡한 논리적 연결을 수행하는 수정 및 확장 작업에는 한계가 있음을 정량적, 정성적으로 규명하였다.

## 📎 Related Works

기존 연구들은 주로 일반적인 프로그래밍 언어를 대상으로 LLM의 코드 생성 및 디버깅 능력을 평가해 왔다. 예를 들어, GitHub Copilot이나 Codex와 같은 모델들이 소프트웨어 개발자의 생산성을 높이는 방향으로 연구되었다. 하지만 본 논문은 다음과 같은 이유로 기존 접근 방식과 Scientific Workflow가 근본적으로 다르다고 주장한다.

1. **도구의 불투명성**: 워크플로우 스크립트는 내부 로직보다 외부 도구(External tools)를 호출하는 형태가 많으며, 도구의 이름이 일반적이지 않아 LLM이 그 기능을 정확히 파악하기 어렵다.
2. **인프라 의존성**: 일반 프로그램과 달리 워크플로우는 분산 시스템의 파일 시스템, 리소스 매니저, 스케줄러 등 하위 인프라 스택에 강하게 의존하며, 이는 일반적인 코드 학습 데이터로는 해결할 수 없는 부분이다.
3. **데이터 부족**: Java나 Python 같은 표준 언어에 비해 Nextflow, Apache Airflow와 같은 워크플로우 시스템의 공개 저장소 데이터는 극히 적어 LLM이 충분히 학습되지 않았을 가능성이 크다.

## 🛠️ Methodology

본 연구는 ChatGPT(GPT-3.5 버전)를 사용하여 세 가지 시나리오에 대한 사용자 연구를 설계하였다.

### 1. 실험 설계 및 구성

- **대상 도메인**: 생물정보학 및 지구 관측 분야.
- **활용 워크플로우**: 총 5종의 실제 워크플로우(Nextflow 기반 4종, Apache Airflow 기반 1종)를 사용하였다.
- **참가자**: 각 분야의 석사 학위 이상을 보유한 도메인 전문가 4명.
- **평가 방법**: 5점 리커트 척도(Likert scale) 설문지, 정량적 정확도 측정(탐지된 태스크 수 등), 실제 실행 가능 여부 확인.

### 2. 세부 연구 단계 (Three User Studies)

#### Study I: 워크플로우 이해 (Workflow Comprehension)

- **목표**: ChatGPT가 주어진 스크립트를 보고 워크플로우의 목적, 개별 태스크, 사용 도구, 입출력 데이터 및 관련 연구 질문을 정확히 설명할 수 있는지 평가한다.
- **절차**: 워크플로우 텍스트를 프롬프트로 제공하고, "Nextflow 개념을 설명하지 말고 워크플로우의 특성만 설명하라"는 제약 조건을 부여하여 응답을 생성하게 한다.

#### Study II: 워크플로우 수정 (Workflow Modification)

- **목표**: 특정 소프트웨어 도구를 다른 대안 도구로 교체하는 능력을 평가한다.
- **절차**:
    1. 특정 단계(예: Read quality control)의 대안 도구 10가지를 추천받는다.
    2. 추천된 도구 중 최적의 도구 2가지를 선정하고 기존 도구와의 차이점 및 장단점을 분석한다.
    3. 선택한 도구를 사용하여 실제 워크플로우 스크립트를 재작성(Rewriting)하게 한다.

#### Study III: 워크플로우 확장 (Workflow Extension)

- **목표**: 일부가 누락된 워크플로우 스크립트를 제공하고, 원래의 연구 목표를 달성하기 위해 필요한 단계를 추가하도록 한다.
- **절차**: 워크플로우의 끝부분 단계를 제거한 후, ChatGPT에게 부족한 단계를 나열하게 하고, 실제 실행 가능한 확장 스크립트를 생성하도록 요청한다. 특히 도구 기반(Tool-based)과 스크립트 기반(Script-based, R/Python) 확장을 구분하여 평가하였다.

## 📊 Results

### 1. 워크플로우 이해 (Study I)

- **정량적 결과**: 연구 분야 및 전반적인 목표 파악에서 높은 정확도를 보였다 ($\mu=4.7, \sigma=0.68$). 대부분의 워크플로우에서 계산 단계와 사용 도구를 정확히 식별하였다.
- **특이사항**: 다만, 독자적으로 구현된 R/Python 스크립트가 많은 워크플로우(WF4-Grasslands)에서는 약어(예: 'fnf'를 'fold and fill'이 아닌 'fraction of non-forest'로 오해)를 잘못 해석하여 성능이 급격히 저하되었다.
- **연구 질문 생성**: 15개 생성 질문 중 10개만이 유효하여, 탐색적 작업(Explorative tasks)에서는 성능이 다소 낮게 나타났다.

### 2. 워크플로우 수정 (Study II)

- **도구 탐색**: 대안 도구 추천 및 일반적인 장단점 설명은 매우 우수하였다 (평균 4.1점).
- **스크립트 생성**: 실제 코드 수정 결과는 매우 부정적이었다. 생성된 4개의 스크립트 중 단 하나만이 구문적으로 유효(Syntactically valid)했으나, 이조차도 의미적으로는 틀린(Semantically incorrect) 수정이었다.
- **Hallucination**: 특정 도구가 지원하지 않는 기능이라고 설명했으나 실제로는 지원하는 기능인 경우(PolyG trimming)가 발견되는 등 LLM의 환각 현상이 관찰되었다.

### 3. 워크플로우 확장 (Study III)

- **단계 나열**: 필요한 단계를 나열하는 작업은 대체로 정확했으나, 제안 내용이 지나치게 일반적(Generic)이라는 비판을 받았다.
- **코드 생성**: 모든 생성 스크립트가 전문가의 개입 없이 실행되지 않았다.
  - **생물정보학(도구 기반)**: 구문 오류(변수명 오류 등)가 있었으나 전문가는 20분 내에 수정 가능했다.
  - **지구 관측(스크립트 기반)**: 존재하지 않는 라이브러리 함수 사용, 병렬화 코드 누락 등 심각한 오류가 많아 20분 내 수정이 불가능했다.

## 🧠 Insights & Discussion

본 연구를 통해 LLM이 Scientific Workflow 개발에서 가지는 강점과 한계가 명확히 드러났다.

**강점 (Knowledge Store로서의 역할)**:
ChatGPT는 방대한 학습 데이터를 바탕으로 도메인 지식을 제공하고, 사용 가능한 소프트웨어 도구를 추천하며, 복잡한 스크립트의 전반적인 목적을 설명하는 '지식 저장소'로서 매우 효율적이다. 이는 전문가가 웹에서 도구를 검색하는 시간을 크게 단축시킬 수 있음을 시사한다.

**한계 (구현 및 정밀도의 부족)**:

1. **정밀한 도구 특성 파악 불가**: LLM은 도구의 일반적인 기능은 알지만, 특정 버전의 데이터 포맷이나 파일 간의 정밀한 연결(Linkage) 방식과 같은 세부적인 특성(Idiosyncrasies)을 반영하지 못한다.
2. **훈련 데이터의 부족**: 연구진이 GitHub를 분석한 결과, 일반 프로그래밍 언어에 비해 워크플로우 시스템의 저장소 수가 현저히 적다. 이는 LLM이 워크플로우의 구조적 패턴을 충분히 학습하지 못했음을 의미한다.
3. **독자적 스크립트 해석의 취약성**: 표준화된 도구가 아닌 사용자 정의 스크립트(Proprietary scripts)의 경우, 변수명이나 약어에 대한 문맥 파악 능력이 떨어져 오답을 생성할 확률이 높다.

**결론적 해석**:
LLM은 워크플로우의 **'무엇을(What)'** 해야 하는지에 대해서는 훌륭한 가이드를 제공하지만, 그것을 **'어떻게(How)'** 구체적으로 코드로 구현할 것인가에 대해서는 여전히 전문가의 세밀한 검토와 수정이 필수적이다.

## 📌 TL;DR

본 논문은 ChatGPT가 과학적 워크플로우의 **이해 및 도구 추천**에는 매우 유용하지만, **실제 실행 가능한 코드 수정 및 확장**에서는 구문 및 의미적 오류가 많아 한계가 있음을 입증하였다. 특히 표준 도구 기반의 워크플로우보다 사용자 정의 스크립트 기반의 워크플로우에서 성능이 낮게 나타났다. 향후 연구에서는 일반 LLM보다 코드 생성에 특화된 모델(Code Llama 등)을 적용하거나, 워크플로우 전용 데이터셋을 통한 미세 조정(Fine-tuning)이 필요할 것으로 보인다.
