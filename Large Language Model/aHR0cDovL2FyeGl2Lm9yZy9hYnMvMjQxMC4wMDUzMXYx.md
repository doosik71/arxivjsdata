# TPI-LLM: Serving 70B-Scale LLMs Efficiently on Low-Resource Edge Devices

Zonghang Li, Wenjiao Feng, Mohsen Guizani, Hongfang Yu (2024)

## 🧩 Problem to Solve

본 논문은 사용자 상호작용 데이터의 개인정보 보호 문제로 인해 거대 언어 모델(LLM)의 추론 환경이 클라우드에서 엣지(Edge) 디바이스로 이동하는 추세에 주목한다. 하지만 엣지 디바이스는 일반적으로 계산 능력, 메모리, 대역폭이 매우 제한적이며, 대다수가 CUDA 지원 GPU 없이 CPU만으로 동작한다.

특히 Llama 3.1-70B와 같은 70B 규모의 모델은 양자화를 적용하더라도 최소 40GB 이상의 메모리가 필요하며, 이는 일반적인 엣지 디바이스의 메모리 용량(4~16GB)을 크게 초과한다. 기존의 파이프라인 병렬 처리(Pipeline Parallelism)는 단일 사용자 시나리오에서 효율성이 떨어지고, 텐서 병렬 처리(Tensor Parallelism)는 빈번한 통신 오버헤드로 인해 성능 저하가 발생한다. 또한, 기존의 모델 오프로딩 방식(예: Accelerate)은 블로킹 I/O(Blocking I/O) 문제로 인해 추론 속도가 극도로 느려지는 한계가 있다. 따라서 본 논문의 목표는 저자원 엣지 디바이스 환경에서 70B 규모의 모델을 효율적으로 서비스할 수 있는 계산 및 메모리 효율적인 텐서 병렬 추론 시스템인 TPI-LLM을 구축하는 것이다.

## ✨ Key Contributions

TPI-LLM의 핵심 아이디어는 텐서 병렬 처리를 기반으로 하되, 엣지 네트워크의 특성과 메모리 제약을 극복하기 위해 통신 알고리즘과 메모리 스케줄링을 최적화하는 것이다.

1.  **Star-based Allreduce 알고리즘**: 엣지 네트워크에서는 대역폭(Bandwidth)보다 링크 지연 시간(Link Latency)이 주된 병목 구간임을 발견하고, 홉(Hop) 수를 최소화하는 스타 기반의 Allreduce 알고리즘을 구현하여 통신 지연을 줄였다.
2.  **Sliding Window Memory Scheduler**: 제한된 메모리 환경에서 OOM(Out-of-Memory)을 방지하기 위해, 필요한 레이어의 가중치만을 동적으로 로드하고 언로드하는 슬라이딩 윈도우 방식의 스케줄러를 도입하였다. 특히, 디스크 I/O 지연 시간을 계산 및 통신 시간과 겹치게(Overlap) 설계하여 I/O 병목을 숨겼다.
3.  **프라이버시 보존 구조**: 사용자 프롬프트와 생성된 시퀀스를 마스터 노드(사용자 디바이스)에만 유지하고, 워커 노드에는 계산을 위한 임베딩 값만을 전달함으로써 원본 데이터의 노출을 방지하였다.

## 📎 Related Works

기존의 LLM 서빙 시스템은 주로 고성능 GPU 클러스터를 대상으로 설계되었으며, 최근 일부 연구가 엣지 환경으로의 적응을 시도하고 있다.

- **Pipeline Parallelism**: 모델을 여러 디바이스에 분할 배치하는 방식이지만, 단일 요청을 처리하는 엣지 시나리오에서는 파이프라인 버블(Pipeline Bubble)이 발생하여 많은 디바이스가 유휴 상태가 되는 비효율성이 존재한다.
- **Tensor Parallelism**: 계산 부하를 분산하여 속도를 높일 수 있으나, 엣지 환경에서는 빈번한 통신이 발생한다. 특히 Galaxy와 같은 기존 시스템은 Ring-based 또는 Tree-based Allreduce를 사용하는데, 이는 엣지 네트워크의 높은 링크 지연 시간으로 인해 오히려 성능이 저하되는 결과를 초래한다.
- **Memory Offloading**: Accelerate와 같은 라이브러리는 CPU 메모리 부족 시 디스크로 데이터를 오프로딩하지만, 블로킹 I/O 방식으로 인해 토큰 생성 속도가 매우 느려지는 문제가 있다.

TPI-LLM은 이러한 한계들을 분석하여, 엣지 환경에서는 텐서 병렬 처리가 더 효율적임을 증명하고, 통신 알고리즘의 단순화와 비동기 메모리 스케줄링을 통해 이를 구체화하였다.

## 🛠️ Methodology

### 전체 시스템 구조
TPI-LLM은 하나의 마스터 노드(사용자 디바이스)와 여러 개의 워커 노드로 구성된다. 마스터 노드는 모델 가중치를 분할하여 배포하고 프롬프트를 처리하며, 워커 노드는 분할된 가중치를 이용해 병렬 계산을 수행한다.

### 상세 추론 절차
1.  **가중치 분산**: 마스터 노드가 사전 학습된 모델 가중치 $W$를 워커 노드의 계산 능력과 메모리 용량에 맞게 분할하여 전달한다.
2.  **입력 처리 및 브로드캐스트**: 마스터 노드가 사용자 프롬프트를 토큰화하고 입력 임베딩 $H^0 = xW_{emb}$를 생성하여 모든 워커 노드에 전송한다.
3.  **텐서 병렬 계산**: 각 워커 노드는 다음과 같은 사이클을 반복한다.
    - **Attention 계산**: 각 노드 $h$는 분할된 가중치 $W_{h,l}^Q, W_{h,l}^K, W_{h,l}^V$를 사용하여 부분적인 Attention 결과물을 계산한다.
    $$H_l^{attn} = \text{allreduce}\left(\text{softmax}\left(\frac{Q_{h,l} (K_{h,l})^\top}{\sqrt{d}}\right) V_{h,l}\right) + H_{l-1}^{ffn}$$
    여기서 $\text{allreduce}$를 통해 모든 노드의 결과를 합산하며, KV 캐시는 각 노드에 로컬로 저장하여 중복 계산을 피한다.
    - **FFN 계산**: 마찬가지로 분할된 FFN 가중치를 사용하여 계산 후 Allreduce를 수행한다.
    $$H_l^{ffn} = \text{allreduce}(W_{h,l}^{down} \cdot (\sigma(W_{h,l}^{gate} \cdot H_l^{norm}) \odot (W_{h,l}^{up} \cdot H_l^{norm}))) + H_l^{attn}$$
4.  **최종 토큰 생성**: 마스터 노드가 최종 결과 $H_L^{ffn}$을 받아 Task Head $W_{head}$와 Softmax를 거쳐 다음 토큰 $z = \text{softmax}(H_L^{ffn} W_{head})$를 샘플링한다.

### Star-based Allreduce
논문은 엣지 네트워크의 병목이 대역폭이 아닌 링크 지연 시간 $\tau$에 있음을 분석하였다. Ring 또는 Tree 알고리즘은 데이터 전송 홉 수가 많아 지연 시간이 증폭된다. 반면 Star-based 알고리즘은 워커가 마스터에게 데이터를 보내고 다시 받는 단순한 구조로, 홉 수를 최소화하여 지연 시간을 $8\tau$ 수준으로 낮추어 가장 효율적임을 보였다.

### Sliding Window Memory Scheduling
메모리 부족 문제를 해결하기 위해 비동기 데몬 스레드가 가중치를 관리한다.
- **작동 방식**: 현재 계산 중인 블록의 다음 블록 가중치를 미리 로드(Preload)하고, 사용이 끝난 가중치는 즉시 언로드(Unload)하여 메모리 점유율을 일정하게 유지한다.
- **정상 상태(Steady State) 조건**: 디스크 I/O 지연 시간이 계산 및 통신 시간에 의해 완전히 가려지기 위한 조건은 다음과 같다.
  $$\text{Tight Steady Condition: } t_{attn} + t_{allreduce} \geq \tau_{ffn} \text{ and } t_{ffn} + t_{allreduce} \geq \tau_{attn}$$
  여기서 $t$는 계산/통신 시간, $\tau$는 가중치 로딩 시간이다. 이 조건이 만족되면 I/O로 인한 계산 중단 없이 추론이 가능하다.

## 📊 Results

### 실험 환경
- **에뮬레이션**: Klonet을 사용하여 8개의 엣지 디바이스(각 8코어 CPU, 8GB RAM) 환경을 구축하였다. 대역폭은 300Mbps, 지연 시간은 1ms로 설정하였다.
- **실제 테스트베드**: 4대의 노트북(MacBook Pro, MacBook Air, Dell)을 Wi-Fi 공유기로 연결하여 테스트하였다.
- **대상 모델**: Llama 2 (3B~70B), Llama 3.1 (8B, 70B), Yi-34B 등.

### 주요 결과
1.  **메모리 효율성**: Llama 2-70B 모델을 8대의 디바이스에서 실행했을 때, 피크 메모리 사용량을 디바이스당 **3.1 GB**까지 낮추어 70B 모델을 저사양 기기에서 구동 가능하게 하였다.
2.  **추론 속도**: 
    - Accelerate 대비 TTFT(Time-to-First-Token) 및 토큰 지연 시간을 **80% 이상** 단축하였다.
    - Transformers 및 Galaxy 대비 **90% 이상**의 성능 향상을 보였다.
3.  **확장성 분석**: 디바이스 수와 CPU 코어 수가 증가함에 따라 지연 시간이 유의미하게 감소하였으나, 네트워크 대역폭을 300Mbps에서 1Gbps로 높이는 것은 성능 향상에 거의 영향을 주지 않았다. 이는 통신 데이터 크기가 매우 작아 대역폭보다 계산 성능이 주된 병목임을 시사한다.
4.  **벤치마크 비교**: Galaxy는 Ring-based 알고리즘을 사용하므로 링크 지연 시간이 누적되어 MP(Model Parallelism)보다도 느린 경우가 발생했다. 반면 TPI-LLM은 Star-based Allreduce와 메모리 스케줄러를 통해 가장 낮은 지연 시간을 기록했다.

## 🧠 Insights & Discussion

TPI-LLM은 엣지 컴퓨팅 환경의 특수성을 정확히 짚어내어 해결책을 제시하였다. 특히, 데이터 센터 환경에서는 당연하게 여겨지는 고대역폭 네트워크 가정이 엣지에서는 성립하지 않으며, 오히려 **링크 지연 시간(Link Latency)**이라는 변수가 통신 알고리즘 선택에 결정적인 영향을 미친다는 점을 정량적으로 분석한 점이 돋보인다.

또한, 단순히 가중치를 나누어 갖는 텐서 병렬 처리를 넘어, CPU 환경에서의 디스크 I/O 병목을 해결하기 위해 **비동기 슬라이딩 윈도우 스케줄링**을 도입함으로써 메모리 제약과 속도라는 두 마리 토끼를 잡았다.

다만, 실험 결과에서 나타나듯 네트워크 대역폭의 영향이 적고 계산 성능(CPU 코어 수)이 여전히 주된 병목으로 남아있다는 점은 향후 연구 과제이다. CPU 기반의 추론 가속화나 엣지 전용 경량 연산 커널의 도입이 병행된다면 더욱 획기적인 성능 향상이 가능할 것으로 보인다.

## 📌 TL;DR

본 논문은 저사양 엣지 디바이스에서 70B 규모의 LLM을 효율적으로 구동하기 위한 시스템 **TPI-LLM**을 제안한다. 엣지 네트워크의 특성인 높은 링크 지연 시간을 해결하기 위해 **Star-based Allreduce**를 도입하고, 메모리 부족 문제를 해결하기 위해 디스크 I/O와 계산을 오버랩하는 **Sliding Window Memory Scheduler**를 구현하였다. 그 결과, Llama 2-70B 모델을 디바이스당 단 3.1GB의 메모리만으로 구동할 수 있었으며, 기존 엣지 추론 프레임워크 대비 지연 시간을 80~90% 이상 획기적으로 줄였다. 이 연구는 향후 개인정보 보호가 중요한 온디바이스 LLM 서비스 구현에 핵심적인 기반 기술이 될 가능성이 높다.
