# Reflection-Driven Control for Trustworthy Code Agents

Bin Wang, Jiazheng Quan, Xingrui Yu, Hansen Hu, Yuhao, Ivor Tsang (2026)

## 🧩 Problem to Solve

현대의 대규모 언어 모델(LLM) 기반 에이전트들은 뛰어난 능력을 갖추고 있으나, 신뢰할 수 있는 안전 제어 장치가 부족하여 제약 없는, 예측 불가능하며, 심지어는 유해한 출력을 생성할 위험이 있다. 특히 도구 사용과 자율성이 결합된 에이전트 환경에서는 탈옥(jailbreak), 프롬프트 주입(prompt injection), 에이전트 웜(agent worms)과 같은 보안 취약점이 시스템 전체의 리스크로 직결될 수 있다.

본 논문은 특히 보안에 민감한 코드 생성(secure code generation) 작업에서 다음과 같은 세 가지 핵심 문제를 해결하고자 한다. 첫째, 추론 및 실행 단계에서 에이전트의 행동을 동적으로 제약하여 작업 이탈과 위험한 도구 호출을 방지하는 결정 시점의 신뢰 제어(trusted control at decision time)이다. 둘째, 결정의 근거와 실행 로직을 추적하여 책임성과 투명성을 보장하는 사후 검증 가능성 및 감사 가능성(post-hoc verifiability and auditability)의 확보이다.

결과적으로 본 연구의 목표는 코드의 보안성(Security), 원래의 의도를 유지하는 기능성(Functionality), 그리고 컴파일 및 실행 가능성(Executability)을 동시에 만족하는 자율적이고 신뢰할 수 있는 코드 생성 에이전트를 구축하는 것이다.

## ✨ Key Contributions

본 논문의 핵심 아이디어는 '자기 성찰(self-reflection)'을 단순한 사후 처리 패치가 아닌, 에이전트 추론 과정의 명시적이고 필수적인 단계로 격상시키는 **Reflection-Driven Control** 프레임워크를 제안하는 것이다.

중심적인 설계 직관은 에이전트 내부에 '성찰 루프(reflection loop)'를 구축하여 자신의 결정 경로를 지속적으로 모니터링하고 평가하게 하는 것이다. 잠재적 위험이 감지되면, 시스템은 진화하는 성찰 메모리(reflective memory)에서 관련 수정 사례와 보안 코딩 가이드라인을 검색하여, 증거 기반의 제약 조건을 다음 추론 단계에 직접 주입한다. 이를 통해 에이전트는 자율성을 유지하면서도 설계 단계부터 안전함이 보장되고(safer by construction), 모든 수정 과정이 기록되어 감사 가능한 상태가 된다.

## 📎 Related Works

기존의 에이전트 프레임워크와 시스템 보안 연구들은 멀티 에이전트 협업을 통한 작업 효율성 증대에 집중해 왔으나, 에이전트 간의 조정 실패나 정렬되지 않은 목표로 인한 시스템적 리스크를 완전히 해결하지 못했다. 특히 THOR 프레임워크와 같은 최근 연구들은 에이전트 생애 주기 전반에 걸친 계층적 방어와 감사 가능성을 강조하고 있다.

또한, TRiSM(Trust, Risk, and Security Management) 프레임워크를 통한 에이전트 보안 연구들이 진행되었으며, 입력 단계의 self-reminder 메커니즘이나 출력 단계의 decoupled refusal training 등이 제안되었다. 그러나 이러한 접근 방식들은 입력과 출력 레이어에서 파편화된 보호를 제공할 뿐, 추론, 계획, 도구 실행을 통합적으로 관리하는 폐쇄 루프(closed-loop) 제어 시스템은 부족한 실정이다. 본 논문은 이러한 파편화된 방어 체계를 넘어, 추론 과정 내부에 성찰 제어 회로를 통합함으로써 기존 연구와의 차별점을 가진다.

## 🛠️ Methodology

### 전체 시스템 구조
본 논문은 $\text{Plan} \rightarrow \text{Reflect} \rightarrow \text{Verify}$로 이어지는 에이전트 프레임워크를 제안하며, 그 중심에 플러그인 형태로 통합 가능한 **Reflex Module**을 배치한다. 이 모듈은 일반적인 에이전트 오케스트레이션에 쉽게 통합될 수 있도록 설계되었다.

### 핵심 구성 요소 및 역할
Reflex Module은 다음과 같은 세 가지 핵심 구성 요소로 이루어져 있다.

1.  **Lightweight Self-Checker**: 입력 코드를 빠르게 스캔하여 "SAFE" 또는 "UNSAFE"로 분류하는 이진 분류기이다. "SAFE"로 판정되면 불필요한 계산을 피하기 위해 즉시 메모리에 저장하고 종료하며, "UNSAFE"일 경우에만 전체 성찰 및 수정 파이프라인을 가동하여 추론 비용을 최적화한다.
2.  **Reflective Prompt Engine**: "UNSAFE" 판정을 받은 코드에 대해 심층 분석을 수행한다. 단일 생성 작업을 다회차의 $\text{Chain-of-Thought}$ 성찰 대화로 변환하여 $\text{취약점 식별} \rightarrow \text{원인 분석 및 완화 전략 수립} \rightarrow \text{보안 코드 수정}$의 단계를 거치게 한다.
3.  **Reflective Memory Repository**: 지속적인 학습과 경험 재사용을 위한 지식 베이스이다. 이는 두 가지 계층으로 구성된다.
    *   **Dynamic Memory ($M_D$)**: 시스템 운영 중 생성되고 검증된 수정 사례를 저장하는 ChromaDB 기반의 벡터 데이터베이스이다.
    *   **Static Memory ($M_S$)**: 표준 보안 코딩 규격 및 취약점 데이터베이스와 같은 기초 지식을 저장한다.

### 주요 방정식 및 절차
보안 코드 생성 작업은 다음과 같은 조건부 생성 프로세스로 공식화된다.

$$y = G(x, C_f, C_{fn}, R(x, C_f, C_{fn}; M), \theta)$$

여기서 $G$는 LLM 생성기, $x$는 보안 결함이 있는 코드, $C_f$와 $C_{fn}$은 각각 파일 및 함수 레벨의 컨텍스트, $R$은 메모리 $M$에서 보안 수정 사례를 추출하는 검색 함수, $\theta$는 모델 및 프롬프트 파라미터를 의미한다.

검색 함수 $\text{Retrieve}(M_D, M_S, q)$의 작동 방식은 다음과 같다.

$$E = \text{Retrieve}(M_D, M_S, q) = \begin{cases} \text{Top-k from } M_D, & \text{if condition} \\ \text{Top-k from } M_D \cup M_S, & \text{otherwise} \end{cases}$$

이때 조건은 $M_D$에서 검색된 결과의 수 $|E_{M_D}|$가 최소 임계값 $k_{min}$ 이상이고, 최대 코사인 유사도가 하한선 $\theta$를 넘을 때만 동적 메모리를 우선 사용하며, 그렇지 않을 경우 정적 메모리 $M_S$를 통해 지식을 보충한다.

## 📊 Results

### 실험 설정
- **데이터셋**: MITRE의 Top 25 위험 소프트웨어 약점을 기반으로 한 8가지 CWE(Common Weakness Enumeration) 카테고리의 챌린징 시나리오를 사용하였다. (예: SQL Injection, Buffer Overflow, XSS 등)
- **측정 지표**:
    - $\text{Security Rate (Sec. Rate)}$: 컴파일 가능한 샘플 중 보안 결함이 없는 비율.
    - $\text{Pass Rate}$: 컴파일 가능 샘플 중 기능적 테스트를 통과한 비율.
    - $\text{Total Efficiency (Eff. Total)}$: 성공적으로 컴파일된 작업의 수.
- **검증 도구**: 정적 분석 도구인 CodeQL과 LLM Judge를 활용하여 정량적/정성적 평가를 수행하였다.

### 주요 결과
1.  **보안성 및 기능성 향상**: GPT-3.5, GPT-4, Qwen-3-Coder, Gemini-2.5-Pro 등 다양한 모델에서 Reflex Module 적용 시 $\text{Security Rate}$가 일관되게 상승하였다. 특히 Qwen 모델의 경우 $83.7\% \rightarrow 94.9\%$로 $11.2\%$p의 큰 폭의 향상을 보였다. $\text{Pass Rate}$는 대부분의 모델에서 유지되거나 소폭 상승하여, 보안 강화가 기능적 정확성을 해치지 않음을 입증하였다.
2.  **동적 RAG의 지식 축적**: 반복적인 실행 횟수가 늘어남에 따라 검색 유사도가 $0.850$에서 $0.980$으로 증가하였으며, 정적 메모리로의 폴백(fallback) 비율이 $15\%$에서 $0\%$로 감소하여 시스템이 자가 강화되는 양상을 보였다.
3.  **성찰 깊이와 한계 효용**: 성찰 횟수가 늘어날수록 성능이 향상되지만, 약 4회차 이후부터는 성능 향상이 정체되는 지점(plateau)이 발견되었다. 특히, 단 한 번의 정교한 성찰 라운드만으로도 핵심 수정 패턴의 약 $90\%$를 포착할 수 있다는 점이 확인되었다.
4.  **비용 및 오버헤드**: 시나리오당 평균 비용은 $5.37 \times 10^{-4}$ 달러로 매우 낮았으며, 전체 실행 시간의 $84.4\%$가 LLM 추론 시간에 집중되어 있어, Reflex 모듈 자체의 논리적 오버헤드(RAG 검색 등)는 무시할 수 있는 수준($2.8\%$)이었다.

## 🧠 Insights & Discussion

본 논문은 LLM 에이전트의 안전성을 확보하기 위해 모델을 재학습시키지 않고도 아키텍처 수준의 제어 루프를 도입함으로써 실무적인 해결책을 제시하였다. 특히 동적 메모리와 정적 메모리를 결합한 계층적 RAG 구조는 최신 수정 사례와 기초 보안 원칙을 동시에 활용하게 함으로써 적응성과 완전성을 동시에 확보하였다.

성찰 과정에서 발생하는 '한계 효용'에 대한 분석은 매우 중요한 통찰을 제공한다. 무한한 성찰보다는 최적의 깊이(예: 1~4회)를 설정하는 것이 계산 비용과 보안 성능 사이의 최적의 균형점임을 시사한다. 이는 향후 실시간 시스템에 적용할 때 추론 지연 시간을 최소화하면서도 보안성을 극대화할 수 있는 가이드라인이 된다.

다만, 본 연구는 특정 CWE 시나리오에 집중되어 있으며, 실제 복잡한 대규모 프로젝트의 전체 코드베이스에서 발생할 수 있는 논리적 취약점이나 아키텍처 수준의 보안 결함까지 모두 해결할 수 있는지는 추가적인 검증이 필요하다. 또한, LLM Judge의 평가 결과가 실제 공격 시나리오에서의 방어력과 얼마나 일치하는지에 대한 정밀한 분석이 보완되어야 할 것이다.

## 📌 TL;DR

본 논문은 LLM 기반 코드 생성 에이전트를 위해 **Reflection-Driven Control**이라는 플러그인 모듈을 제안한다. 이 모듈은 $\text{가벼운 자가 점검} \rightarrow \text{다회차 성찰 추론} \rightarrow \text{계층적 메모리(RAG) 기반 수정}$의 폐쇄 루프를 통해, 모델의 재학습 없이도 생성된 코드의 보안성과 감사 가능성을 획기적으로 높인다. 실험 결과, 다양한 LLM에서 보안율이 크게 향상되었으며, 추가적인 시간 및 비용 오버헤드는 매우 적어 실제 고위험 소프트웨어 개발 환경에 적용 가능성이 매우 높다.
