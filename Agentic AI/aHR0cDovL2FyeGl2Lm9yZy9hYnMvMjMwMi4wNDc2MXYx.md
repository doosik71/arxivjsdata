# Toolformer: Language Models Can Teach Themselves to Use Tools

Timo Schick, Jane Dwivedi-Yu, Roberto Dessì, Roberta Raileanu, Maria Lomeli, Luke Zettlemoyer, Nicola Cancedda, Thomas Scialom (2023)

## 🧩 Problem to Solve

거대 언어 모델(Large Language Models, LMs)은 대규모 데이터 학습을 통해 새로운 태스크에 대해 뛰어난 제로샷(zero-shot) 및 퓨샷(few-shot) 능력을 보여주지만, 역설적으로 단순한 산술 연산이나 사실 관계 확인과 같은 기본적인 기능에서는 어려움을 겪는다. 이러한 문제는 모델의 크기를 단순히 키우는 것(scaling)만으로는 완전히 해결하기 어려우며, 최신 정보에 접근하지 못해 발생하는 환각(hallucination) 현상, 저자원 언어 이해 부족, 수학적 정밀도 결여, 시간의 흐름에 대한 인식 부재 등의 내재적 한계로 이어진다.

기존에 이러한 한계를 극복하기 위해 외부 도구(search engines, calculators 등)를 사용하는 방식이 제안되었으나, 이는 대량의 인간 주석(human annotations)에 의존하거나 특정 태스크에만 국한된 설정에서 작동한다는 단점이 있다. 따라서 본 논문의 목표는 인간의 대규모 주석 없이도 언어 모델이 스스로 어떤 API를, 언제, 어떻게 호출해야 하는지를 학습하여 일반적인 언어 모델링 능력을 유지하면서도 외부 도구를 효율적으로 사용할 수 있게 하는 모델인 Toolformer를 개발하는 것이다.

## ✨ Key Contributions

본 논문의 핵심 아이디어는 언어 모델의 In-context learning 능력을 활용하여 스스로 API 호출 데이터를 생성하고, 이 중 실제로 모델의 예측 성능을 향상시키는 호출만을 선별하여 학습하는 self-supervised 방식의 도구 사용 학습법이다. 모델은 소수의 API 사용 예시만을 제공받은 후, 거대한 텍스트 데이터셋 내에서 잠재적인 API 호출 위치를 샘플링하고 실행한다. 이후, API의 결과값이 다음 토큰을 예측하는 데 실제로 도움이 되었는지를 손실 함수(loss function)를 통해 정량적으로 평가하여 필터링하며, 최종적으로 유용한 API 호출이 포함된 데이터셋으로 모델을 파인튜닝함으로써 도구 사용 능력을 습득한다.

## 📎 Related Works

기존의 연구들은 크게 세 가지 방향으로 진행되었다. 첫째, 프리트레이닝 단계에서 메타데이터나 HTML 태그, 검색 결과 등을 함께 제공하는 방식이다. 그러나 이러한 방식은 정보가 유용한지 여부와 관계없이 항상 제공된다는 한계가 있다. 둘째, 검색 엔진이나 계산기 등의 외부 도구를 사용하는 방식인데, 이는 주로 대량의 인간 감독 데이터에 의존하거나 특정 태스크를 위한 퓨샷 프롬프팅에 의존한다. 셋째, self-training이나 bootstrapping 기술을 통해 모델을 개선하는 방식이다.

Toolformer는 이러한 기존 연구들과 달리, 특정 태스크에 종속되지 않고 모델이 스스로 필요에 따라 도구를 호출하는 시점과 방법을 결정한다는 점에서 차별성을 가진다. 또한, 인간의 개입을 최소화하고 모델 자신의 피드백(perplexity 감소)을 통해 학습 데이터를 구축한다는 점에서 효율적이다.

## 🛠️ Methodology

### 전체 파이프라인
Toolformer의 학습 과정은 크게 세 단계의 데이터 구축 과정과 최종 파인튜닝 단계로 구성된다.

1.  **API 호출 샘플링 (Sampling API Calls):** In-context learning을 위해 API 사용 예시가 포함된 프롬프트 $P(x)$를 작성한다. 모델 $M$은 입력 텍스트 $x$의 각 위치 $i$에서 API 호출을 시작할 확률 $p_i = p_M(\langle \text{API} \rangle | P(x), x_{1:i-1})$를 계산하며, 임계값 $\tau_s$보다 높은 상위 $k$개의 위치를 선정한다. 각 위치에서 모델은 $\langle \text{API} \rangle$ 토큰 이후의 API 호출 내용 $c$를 생성한다.
2.  **API 호출 실행 (Executing API Calls):** 생성된 API 호출 $c$를 실제로 실행하여 결과값 $r$을 얻는다. 사용하는 도구에 따라 파이썬 스크립트 실행, 검색 엔진 쿼리, 다른 신경망 모델 호출 등이 이루어진다.
3.  **API 호출 필터링 (Filtering API Calls):** 얻어진 결과값이 실제로 다음 토큰 예측에 도움이 되었는지 평가한다.

### 손실 함수 및 필터링 기준
필터링을 위해 가중 교차 엔트로피 손실(weighted cross entropy loss) $L_i$를 사용한다. 위치 $i$에서의 손실 함수는 다음과 같이 정의된다.
$$L_i(z) = -\sum_{j=i}^{n} w_{j-i} \cdot \log p_M(x_j | z, x_{1:j-1})$$
여기서 $z$는 모델에 제공되는 접두어(prefix)이며, $w$는 API 호출 시점과 정보 활용 시점의 거리를 가깝게 유지하기 위한 가중치 함수이다.

모델은 다음 두 가지 손실 값을 비교한다.
- $L_i^+$: API 호출과 그 결과값 $e(c_i, r_i)$가 모두 제공되었을 때의 손실
- $L_i^-$: API 호출을 하지 않았을 때($\epsilon$)와 API 호출은 했으나 결과값은 제공되지 않았을 때($e(c_i, \epsilon)$) 중 더 작은 손실 값
$$L_i^- = \min(L_i(\epsilon), L_i(e(c_i, \epsilon)))$$

API 호출이 유용하다고 판단되는 기준은 $L_i^- - L_i^+ \geq \tau_f$일 때이다. 즉, API 결과값이 제공되었을 때의 손실이 제공되지 않았을 때보다 임계값 $\tau_f$ 이상으로 감소해야만 해당 호출을 유지한다.

### 모델 파인튜닝 및 추론
필터링을 통해 선택된 유용한 API 호출들을 원문 텍스트에 삽입하여 확장된 데이터셋 $C^*$를 구축한다. 이후 표준 언어 모델링 목적 함수를 사용하여 $M$을 $C^*$로 파인튜닝한다.

추론 시에는 모델이 $\rightarrow$ 토큰을 생성하면 디코딩을 일시 중단하고, 해당 API를 호출하여 결과를 얻은 뒤, 그 결과를 텍스트에 삽입하고 다시 디코딩을 재개하는 방식으로 동작한다.

## 📊 Results

### 실험 설정
- **기본 모델:** GPT-J (6.7B parameters)
- **사용 도구:** QA 시스템(Atlas), 계산기, 위키피디아 검색(BM25), 기계 번역(NLLB), 캘린더 API
- **비교 대상:** 기본 GPT-J, CCNet으로 파인튜닝한 GPT-J, Toolformer (API 사용 불가 설정), OPT (66B), GPT-3 (175B)
- **평가 지표:** 제로샷 설정에서의 정확도 및 언어 모델링 성능(Perplexity)

### 주요 결과
1.  **지식 기반 및 수학 태스크:** LAMA 벤치마크와 수학 데이터셋(ASDiv, SVAMP, MAWPS)에서 Toolformer는 동일 크기의 모델뿐만 아니라 훨씬 거대한 GPT-3 (175B)보다 뛰어난 성능을 보였다. 특히 수학 태스크에서는 계산기 도구를 97.9%의 확률로 호출하여 성능을 극대화했다.
2.  **질의응답 (QA):** WebQuestions, Natural Questions 등에서 기본 모델보다 향상된 성능을 보였으나, GPT-3 (175B)에는 미치지 못했다. 이는 단순한 검색 엔진의 한계와 상호작용(interactive) 능력 부족으로 해석된다.
3.  **다국어 및 시간 인식 태스크:** MLQA(다국어 QA)에서 번역 도구를 활용해 성능을 높였으며, DATESET과 같은 시간 관련 태스크에서 캘린더 API를 통해 압도적인 성능 향상을 보였다.
4.  **일반 능력 유지:** WikiText와 CCNet 데이터셋에 대한 Perplexity 측정 결과, API 호출 학습 후에도 일반적인 언어 모델링 능력이 저하되지 않았음을 확인했다.

## 🧠 Insights & Discussion

### 강점 및 분석
Toolformer는 외부 도구 사용 능력을 학습시키기 위해 막대한 양의 사람이 작성한 데이터가 필요하지 않다는 점이 가장 큰 강점이다. 또한, 모델 규모에 따른 분석 결과, 도구 사용 능력이 약 775M 파라미터 규모의 모델부터 발현(emergence)된다는 점을 발견했다. 이는 모델이 어느 정도의 기초 능력을 갖춰야만 API 호출이라는 복잡한 메커니즘을 학습할 수 있음을 시사한다.

### 한계 및 비판적 해석
본 논문에서 제시된 방식은 몇 가지 명확한 한계가 존재한다. 
첫째, **도구 체이닝(Tool Chaining)**이 불가능하다. 한 도구의 출력값을 다른 도구의 입력값으로 사용하는 연속적인 호출 구조를 학습하지 못했다. 
둘째, **비상호작용적(Non-interactive) 호출**이다. 검색 결과가 만족스럽지 않을 때 쿼리를 수정하거나 여러 결과를 브라우징하는 능력이 없다. 
셋째, **데이터 효율성** 문제이다. 수백만 개의 문서를 처리해도 계산기 API와 같이 유용한 사례가 적은 도구의 경우, 학습 샘플 수가 매우 적게 추출되는 경향이 있다.

## 📌 TL;DR

Toolformer는 언어 모델이 스스로 어떤 API를 호출해야 할지 학습하는 self-supervised 프레임워크를 제안한다. 모델은 In-context learning으로 API 호출 후보를 생성하고, 해당 호출이 다음 토큰 예측의 perplexity를 낮추는지를 기준으로 유용한 데이터만을 필터링하여 학습한다. 이를 통해 6.7B 규모의 모델이 특정 영역(수학, 사실 확인 등)에서 175B 규모의 GPT-3를 능가하는 성능을 보였으며, 일반적인 언어 능력의 손실 없이 도구 사용 능력을 획득할 수 있음을 입증했다. 이 연구는 향후 LLM이 단순한 텍스트 생성을 넘어 외부 환경과 상호작용하는 Agent로 발전하는 데 중요한 기여를 할 것으로 보인다.
