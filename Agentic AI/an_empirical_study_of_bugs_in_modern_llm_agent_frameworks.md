# An Empirical Study of Bugs in Modern LLM Agent Frameworks

Xinxue Zhu, Jiacong Wu, Xiaoyu Zhang, Tianlin Li, Yanzhou Mu, Juan Zhai, Chao Shen, Chunrong Fang, and Yang Liu (2026)

## 🧩 Problem to Solve

본 논문은 현대의 LLM 에이전트 프레임워크(LLM Agent Frameworks)에서 발생하는 버그의 특성을 분석하고 체계화하는 것을 목표로 한다. LLM 에이전트는 메모리, 도구, 제어 로직을 통합하여 복잡한 작업을 수행하며, 이를 위해 LangChain이나 CrewAI와 같은 프레임워크에 크게 의존한다.

문제의 핵심은 기존 연구들이 주로 에이전트 수준의 실패(추론, 계획, 행동 단계의 오류)에 집중했을 뿐, 그 하단에서 동작하는 프레임워크 수준의 버그(Framework-level bugs)는 간과했다는 점이다. 프레임워크 단계에서 발생한 버그는 상위 시스템으로 전파되어 잘못된 실행, 자원 오용, 심지어 보안 위협으로 이어질 수 있으며, 이는 LLM 소프트웨어 공급망 전체의 안정성을 저해한다. 따라서 본 연구는 에이전트의 동적 실행 생애주기(Dynamic Agent Lifecycle) 관점에서 버그의 근본 원인과 증상을 분석하여, 프레임워크의 품질을 높이기 위한 기초 자료를 제공하고자 한다.

## ✨ Key Contributions

본 논문의 가장 중심적인 기여는 LLM 에이전트의 동작을 다섯 가지 생애주기 단계인 'Agent Initialization(에이전트 초기화)', 'Perception(인지)', 'Self-Action(자기 행동)', 'Mutual Interaction(상호 작용)', 'Evolution(진화)'으로 정의하고, 이를 기반으로 버그를 분석하는 **생애주기 중심의 관점(Lifecycle-oriented perspective)**을 도입한 것이다.

이를 통해 연구진은 CrewAI와 LangChain에서 수집한 998개의 버그 리포트를 분석하여 15가지의 근본 원인(Root Causes)과 7가지의 관찰 가능한 증상(Symptoms)으로 구성된 체계적인 분류 체계(Taxonomy)를 구축하였다. 특히 프레임워크 버그가 단순히 인터페이스의 문제가 아니라 실행 시맨틱(Execution Semantics)과 API 진화 과정에서 주로 발생한다는 점을 밝혀냈다.

## 📎 Related Works

기존의 연구들은 주로 LLM 에이전트가 계획을 잘못 세우거나 추론에 실패하는 '에이전트 수준의 실패'를 행동 분석이나 벤치마크 평가를 통해 다루어 왔다. 하지만 이는 모델의 한계를 이해하는 데는 도움이 되나, 프레임워크 자체의 구현 오류로 인한 문제는 다루지 못한다.

최근 일부 연구에서 에이전트 라이브러리의 버그를 분석하기 위해 풀 리퀘스트(Pull Request)를 정적 구성 요소(예: 데이터 전처리)에 매핑하는 시도가 있었으나, 이는 에이전트의 동적인 실행 흐름과 시간적 워크플로우를 반영하지 못한다는 한계가 있다. 본 논문은 이러한 정적 분석의 한계를 넘어 에이전트의 실행 생애주기에 따라 버그가 어떻게 발현되는지를 분석함으로써 기존 접근 방식과 차별화를 둔다.

## 🛠️ Methodology

본 연구는 다음과 같은 3단계 방법론을 통해 에이전트 프레임워크의 버그를 분석한다.

### 1. 이슈 수집 (Issue Collection)

가장 대표적인 오픈소스 에이전트 프레임워크인 **LangChain**과 **CrewAI**를 대상으로 GitHub에서 이슈 리포트를 수집하였다. LangChain은 도구 호출 및 실행 파이프라인의 풍부한 추상화를 제공하며, CrewAI는 역할 기반의 다중 에이전트 협업에 특화되어 있어 서로 보완적인 분석이 가능하다. 2023년 12월부터 2026년 1월까지 수집된 총 2,773개의 원본 이슈 리포트를 확보하였다.

### 2. 버그 필터링 (Bug Filtering)

수집된 데이터에서 분석 대상이 아닌 항목을 제거하기 위해 2단계 필터링을 수행한다.

- **1단계 (Label Filtering):** 메인테이너가 'bug' 레이블을 부여한 이슈만 1차적으로 선별하여 1,010개를 남겼다.
- **2단계 (Manual Inspection):** 두 명의 연구자가 수동으로 검토하여 단순 오타(Documentation typos), 사용법 질문(Usage questions), 프레임워크 로직과 무관한 인프라 문제 등을 제외하였다. 최종적으로 998개의 정제된 버그 데이터셋을 구축하였다.

### 3. 개별 레이블링 (Individual Labeling)

두 명의 작성자가 독립적으로 각 리포트에 대해 대상 컴포넌트, 근본 원인, 증상을 레이블링하였다.

- **초기 분류 체계 구축:** 전체 데이터 중 무작위로 100개를 샘플링하여 초기 Taxonomy를 설계하고, 합의 과정을 통해 15개의 근본 원인과 7개의 증상 카테고리를 도출하였다.
- **대규모 어노테이션:** 구축된 Taxonomy를 바탕으로 나머지 이슈들에 대해 레이블링을 수행하였으며, 기존 카테고리에 맞지 않는 경우에만 새로운 카테고리를 추가하는 방식으로 일관성을 유지하였다.

## 📊 Results

### RQ1: 에이전트 프레임워크 버그의 근본 원인은 무엇인가?

분석 결과, 15가지의 근본 원인이 도출되었으며, 그 중 **API Misuse (32.97%)**와 **API Incompatibility (22.34%)**가 전체의 55% 이상을 차지하여 가장 지배적인 원인으로 나타났다.

- **API Misuse:** 개발자가 프레임워크의 실행 추상화나 제어 흐름 설계를 오해하여 도구, 메모리, 에이전트 상호작용을 잘못 오케스트레이션하는 경우이다.
- **API Incompatibility:** 프레임워크 API의 변경으로 인해 계획, 도구 호출 등의 실행 시맨틱이 변하여 기존 에이전트 로직의 가정이 깨지는 경우이다.
- **Documentation Desync:** 문서화된 내용과 실제 프레임워크 동작 사이의 불일치로 인해 개발자가 잘못된 API 계약을 사용하는 경우이다.

특히, 이러한 버그들은 **'Self-Action' 단계**에 가장 집중되어 나타났으며, 이는 도구 호출 및 그래프 실행과 같은 실행 시맨틱 메커니즘이 프레임워크 실패의 주된 원인임을 시사한다.

### RQ2: 에이전트 프레임워크 버그의 증상은 어떻게 나타나는가?

7가지의 증상 카테고리가 식별되었으며, 가장 빈번하게 발생하는 증상은 다음과 같다.

- **Functional Error (781/998):** 시스템은 계속 작동하지만 상태 추적, 계획 흐름, 도구 협업 등이 잘못되어 결국 잘못된 결과물을 생성하는 논리적 버그이다.
- **Crash (100/998):** 런타임에 내부 컴포넌트가 실패하여 프레임워크가 예상치 못하게 종료되는 현상이다.
- **Build Failure (67/998):** 설치, 설정 또는 시작 프로세스에서 문제가 발생하여 에이전트가 작동 상태로 진입하지 못하는 경우이다.

증상 역시 **'Self-Action' 단계**에서 가장 다양하고 빈번하게 발생하며, 이는 해당 단계의 버그가 워크플로우 진행을 직접적으로 방해하거나 잘못된 작업 결과를 초래하기 때문이다.

## 🧠 Insights & Discussion

본 연구는 LLM 에이전트 프레임워크의 버그가 단순한 구현 실수가 아니라, **API의 빠른 진화와 실행 시맨틱의 복잡성**에서 기인한다는 점을 실증적으로 보여주었다. 특히 'Self-Action' 단계에서의 높은 버그 집중도는 에이전트가 도구를 호출하고 계획을 실행하는 동적 단계의 안정성 확보가 최우선 과제임을 의미한다.

**강점 및 시사점:**

- 단순한 정적 분석을 넘어 에이전트의 생애주기라는 동적 관점을 도입하여 버그의 전파 경로를 체계화하였다.
- API 오용과 불일치가 주된 원인이라는 점은, 프레임워크 설계 시 더 엄격한 API 계약(Contract) 관리와 표준화된 인터페이스 설계가 필요함을 시사한다.

**한계 및 향후 과제:**

- 분석 대상이 CrewAI와 LangChain 두 가지 프레임워크에 국한되어 있어, 모든 에이전트 프레임워크에 일반화하기에는 한계가 있을 수 있다.
- 향후 연구에서는 본 논문에서 제안한 생애주기 기반의 분석법을 활용하여, 실행 시맨틱 버그를 조기에 탐지할 수 있는 생애주기 인식 테스트(Lifecycle-aware testing) 방법론을 개발할 필요가 있다.

## 📌 TL;DR

본 논문은 LangChain과 CrewAI의 버그 리포트 998개를 분석하여, LLM 에이전트 프레임워크 버그를 5가지 생애주기 단계, 15가지 근본 원인, 7가지 증상으로 분류한 체계(Taxonomy)를 제안한다. 분석 결과, 대부분의 버그는 **'Self-Action' 단계**에서 발생하며, 특히 **API 오용(API Misuse)**과 **API 불일치(API Incompatibility)**가 핵심 원인임이 밝혀졌다. 이 연구는 향후 LLM 에이전트 인프라의 안정성을 높이기 위한 표준 API 설계 및 런타임 모니터링 연구에 중요한 기초 자료가 될 것이다.
