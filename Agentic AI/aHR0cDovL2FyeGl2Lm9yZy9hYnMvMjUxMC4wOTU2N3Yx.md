# Safe, Untrusted, “Proof-Carrying” AI Agents: toward the agentic lakehouse

Jacopo Tagliabue, Ciro Greco (2025)

## 🧩 Problem to Solve

본 논문은 데이터 레이크하우스(Data Lakehouse) 환경에서 AI 에이전트를 통한 자동화를 도입할 때 발생하는 신뢰성, 정확성 및 거버넌스 문제를 해결하고자 한다. 데이터 레이크하우스는 민감한 프로덕션 데이터를 다루는 환경이므로, 자율적인 의사결정을 내리는 AI 에이전트가 직접 데이터 생명주기를 관리하게 될 경우 다음과 같은 치명적인 위험이 존재한다.

첫째, 에이전트가 권한이 없는 데이터에 접근하거나 유출할 가능성이 있는 신뢰성(Trust) 문제이다. 둘째, 검증되지 않은 외부 패키지를 실행하거나 악성 리소스에 접근할 수 있는 보안 위협이 존재한다. 셋째, 에이전트의 실수나 잘못된 코드 생성으로 인해 프로덕션 데이터가 오염되거나 시스템이 중단될 수 있는 정확성(Correctness) 문제이다.

따라서 본 연구의 목표는 "설계 단계부터 안전한(safe-by-design)" 에이전트 워크플로우를 가능하게 하는 프로그래밍 가능한 레이크하우스(Programmable Lakehouse) 추상화를 제안하고, 이를 통해 신뢰할 수 없는(untrusted) 에이전트가 프로덕션 데이터 상에서 안전하게 데이터 파이프라인을 복구하는 프로토타입을 구현하는 것이다.

## ✨ Key Contributions

본 논문의 핵심 아이디어는 데이터 레이크하우스의 모든 구성 요소를 API 기반의 코드로 추상화하여 에이전트가 다루기 쉬운 인터페이스를 제공함과 동시에, 인프라 수준에서 격리와 검증 메커니즘을 구축하는 것이다.

가장 중심적인 설계 아이디어는 **'Git-for-Data'** 개념을 실행 단계에 도입한 것과 **'Verify-then-merge'** 프로토콜을 적용한 것이다. 에이전트의 모든 쓰기 작업은 격리된 브랜치(Branch)에서 수행되며, 결정론적인 검증 함수(Verifier)를 통과한 경우에만 프로덕션(Main) 브랜치로 원자적(Atomic)으로 병합된다. 이를 통해 에이전트에게는 넓은 자율성을 부여하면서도, 실제 프로덕션 데이터에 미치는 영향은 엄격히 통제하는 구조를 설계하였다.

## 📎 Related Works

논문은 기존의 데이터 레이크하우스 아키텍처가 스토리지와 컴퓨팅의 분리, 다중 언어 지원 등의 이점을 가지고 있다고 언급한다. 또한, LLM이 도구(Tool)를 사용하는 능력의 향상으로 인해 자율적인 에이전트의 가능성이 열렸음을 전제한다.

기존의 데이터 관리 방식이나 Snowflake와 dbt 같은 전통적인 스택과의 차별점은 '프로그래밍 가능성(Programmability)'의 수준에 있다. 기존 시스템들은 서로 다른 인터페이스(SQL 에디터, 노트북, 클라우드 웨어하우스 등) 사이를 오가는 컨텍스트 스위칭이 필요하며, 이는 에이전트가 학습하고 조작하기에 매우 복잡한 환경이다. 반면, 본 논문에서 제안하는 방식은 모든 라이프사이클을 코드로 노출하여 에이전트에게 익숙한 '코드'라는 공통 언어(Lingua Franca)를 인터페이스로 제공한다.

## 🛠️ Methodology

### 1. 프로그래밍 가능한 레이크하우스 (Programmable Lakehouse)
본 시스템은 데이터, 인프라 관리, 쿼리 실행, 관측성(Observability)의 모든 과정을 서버 사이드 API, SDK, CLI를 통해 코드로 제어한다.

- **파이프라인 정의**: 파이프라인은 변환(Transformation)의 유향 비순환 그래프(DAG)로 정의된다. 각 변환은 $\text{Table(s)} \to \text{Table}$ 형태의 단순한 함수로 표현되며, 이는 Function-as-a-Service (FaaS) 추상화를 통해 서버리스 런타임에서 효율적으로 실행된다.
- **선언적 I/O 및 인프라**: 사용자는 입력 테이블과 출력 결과물만을 선언적으로 지정하며, 물리적인 I/O 작업은 플랫폼이 직접 수행하여 함수 내부에서 물리적 스토리지(S3 등)에 직접 접근하는 것을 차단한다.

### 2. 트랜잭션 실행 및 Git-for-Data
에이전트가 파이프라인을 실행할 때, 시스템은 Copy-on-Write 기반의 브랜치 전략을 사용한다.

- **Branch-then-Merge**: 실행이 시작되면 자동으로 복제 브랜치가 생성된다. 모든 중간 결과물과 최종 테이블은 이 브랜치에 먼저 기록된다.
- **원자적 병합**: 실행이 완전히 성공하고 검증을 통과한 경우에만 $\text{Branch} \to \text{Main}$ 병합이 원자적으로 수행된다. 만약 실행 중 오류가 발생하면 병합이 일어나지 않으므로, 다운스트림 시스템은 오염된 데이터를 읽지 않고 이전의 일관된 상태를 유지한다.

### 3. 안전성 체크리스트 및 검증 메커니즘
에이전트의 위험을 제어하기 위해 다음과 같은 매핑 구조를 가진다.

| 구분 | 우려 사항 | 적용된 추상화 (Abstraction) |
| :--- | :--- | :--- |
| **신뢰성 (Trust)** | 데이터 무단 접근 | 선언적 I/O (Declarative I/O) |
| **신뢰성 (Trust)** | 악성 코드 실행 | FaaS 런타임 격리 |
| **정확성 (Correctness)** | 프로덕션 데이터 훼손 | 트랜잭션 실행 (Transactional runs) |
| **정확성 (Correctness)** | 잠재적 버그 배포 | 검증 후 병합 (Verify-then-merge) |

특히, 코드의 정확성을 위해 $\text{Branch} \to \text{bool}$ 형태의 검증 함수(Verifier)를 도입한다. 이는 Proof-carrying code 개념에서 영감을 얻은 것으로, 에이전트가 생성한 결과물이 비즈니스 조건 및 안전성 기준을 만족하는지 결정론적으로 확인하는 단계이다.

### 4. self-repairing 에이전트 루프
구현된 프로토타입은 다음과 같은 구성 요소로 이루어진 ReAct 루프를 따른다.
- **Reasoning**: LLM (OpenAI, Anthropic 등)이 추론을 수행한다.
- **Tools**: Bauplan MCP(Model Context Protocol) 서버를 통해 레이크하우스 API(로그 조회, 테이블 쿼리, 브랜치 생성, 실행)를 도구로 사용한다.
- **Outer Loop**: 에이전트가 수정한 파이프라인이 생성한 브랜치를 대상으로 Verifier가 최종 승인을 내리면 Main 브랜치에 병합한다.

## 📊 Results

### 실험 설정
- **데이터셋**: NYC Taxi 데이터셋을 사용하여 두 개의 소스 테이블로부터 두 개의 결과 테이블($A, B$)을 생성하는 간단한 DAG 파이프라인을 구축하였다.
- **시나리오**: NumPy 2.0 릴리스로 인해 pandas 2.0을 사용하는 컨테이너에서 충돌이 발생하는 상황을 시뮬레이션하여 파이프라인을 고의로 고장 냈다.
- **측정 지표**: 정량적 벤치마크보다는 시스템의 동작 가능성(Feasibility)과 안전성 확인에 집중하였다.

### 결과 분석
- **수행 능력**: Sonnet 4.5와 같은 최신 모델은 로그를 검색하고, 레이크의 상태를 쿼리하며, 디버그 브랜치를 생성하여 코드를 수정한 뒤 안전하게 실행하는 전체 과정을 성공적으로 수행하였다.
- **안전성 검증**: GPT-4o-mini와 같이 성능이 낮은 모델의 경우 수리에 실패하였으나, 시스템 구조상 에이전트가 잘못된 조작을 하더라도 프로덕션 데이터에는 아무런 영향이 없었으며 시스템 중단이 발생하지 않았다.
- **비교 분석**: Snowflake와 dbt 같은 기존 스택 역시 MCP 서버를 제공할 수 있으나, 본 논문에서 제안한 수준의 깊은 프로그래밍 가능성과 트랜잭션 브랜칭 구조가 없으므로 동일한 수준의 '안전한 자율 복구'를 지원하기 어렵다고 분석한다.

## 🧠 Insights & Discussion

본 논문은 AI 에이전트가 인프라를 관리하기 위해서는 단순한 API 제공을 넘어, **'에이전트 우선(Agent-first)'** 관점의 시스템 설계가 필요함을 시사한다. 

**강점 및 통찰**:
- 코드를 유니버설 인터페이스로 설정함으로써 LLM의 코드 생성 능력을 인프라 제어에 직접 활용할 수 있게 하였다.
- '신뢰할 수 없는 에이전트'를 전제로 하여, 검증 책임(Proof)을 에이전트에게 부여하고 확인은 결정론적인 시스템(Verifier)이 수행하게 함으로써 보안과 자율성의 균형을 맞추었다.

**한계 및 향후 과제**:
- 본 연구는 프로토타입 수준의 가능성 증명에 집중하였으며, 대규모 벤치마크를 통한 정량적 평가가 부족하다.
- 에이전트가 대규모로 데이터를 탐색하고 실험할 때 발생하는 OLAP 시스템의 병렬 처리 성능 문제가 향후 해결해야 할 주요 과제로 남아 있다.

## 📌 TL;DR

본 논문은 AI 에이전트가 프로덕션 데이터 환경에서 안전하게 동작할 수 있도록 하는 **'프로그래밍 가능한 레이크하우스(Programmable Lakehouse)'** 아키텍처를 제안한다. FaaS 기반의 격리된 실행 환경과 Git-like 데이터 브랜칭, 그리고 결정론적인 검증 함수($\text{Branch} \to \text{bool}$)를 결합하여, 에이전트가 실수하더라도 실제 데이터에는 영향이 없는 **'Safe-by-design'** 구조를 구현하였다. 이를 통해 데이터 파이프라인의 자율 복구 가능성을 증명하였으며, 향후 완전한 자율형 레이크하우스로 나아가는 설계 방향성을 제시하였다.
