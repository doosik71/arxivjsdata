# Call Graph Profiling for Multi Agent Systems

Dinh Doan Van Bien, David Lillis and Rem W. Collier (2015)

## 🧩 Problem to Solve

Multi Agent System (MAS)은 느슨하게 결합된 자율적인 소프트웨어 엔티티들이 상호작용하며 목표를 달성하는 복잡한 시스템이다. 이러한 시스템의 설계, 구현 및 테스트는 매우 까다로운 작업이며, 특히 시스템의 동작 분석, 성능 및 효율성을 진단하기 위한 프로파일링(Profiling) 도구가 다른 프로그래밍 패러다임에 비해 현저히 부족한 실정이다.

일반적인 프로그램에서는 소수의 코드 영역(Bottlenecks 또는 Hot spots)이 전체 실행 시간의 대부분을 차지한다는 점에 착안하여 프로파일링을 수행하지만, MAS는 비동기적인 메시지 전달(Message Passing) 기반의 통신 구조를 가지므로 기존의 함수 호출 기반 프로파일러를 그대로 적용하기 어렵다. 따라서 본 논문의 목표는 MAS의 특성을 반영하여 에이전트 간의 통신과 성능 정보를 연계해 보여줄 수 있는 AgentSpotter의 Call Graph View를 제안하고 구현하는 것이다.

## ✨ Key Contributions

본 논문의 핵심 기여는 전통적인 프로그래밍의 Call Graph 개념을 MAS 환경으로 확장하여, 에이전트 간의 메시지 전달이 시스템의 처리 요구량에 미치는 영향을 정량화하고 이를 계층적 트리 구조로 시각화한 것이다. 특히, '함수 실행 시간'이라는 전통적인 지표를 MAS의 '에이전트 메시지 영향력(Agent Message Impact)'이라는 개념으로 매핑하여, 어떤 에이전트의 어떤 메시지가 시스템 성능의 병목 지점이 되는지를 직관적으로 파악할 수 있게 하였다.

## 📎 Related Works

전통적인 프로파일링 분야에서는 Knuth의 연구 이후 $\text{gprof}$와 같은 도구가 널리 사용되어 왔으며, Java 환경에서는 $\text{JVMTI}$나 $\text{ATOM}$과 같은 API가 존재한다. MAS 분야에서도 일부 도구들이 개발되었으나 다음과 같은 한계가 존재한다.

- **Cougaar 및 LS/TS**: 시스템 모니터링 데이터와 ACL 메시지 데이터를 제공하지만, 프로파일러가 제공하는 것과 같은 의미 있는 합성 지표(Synthetic indicators)를 생성하는 사후 처리 과정이 부족하다.
- **Agent Factory Debugger 및 Brahms Agent Viewer**: 주로 개별 에이전트의 정신 상태나 타임라인을 보여주는 디버깅 도구에 가깝다.
- **Zeus, JADE Sniffer, ACLAnalyser**: 에이전트 간의 상호작용 및 통신 토폴로지를 시각화하여 협력 및 조정 오류를 수정하는 데 도움을 주지만, 성능 분석(Performance Analysis)보다는 디버깅에 초점이 맞춰져 있다.

본 연구는 이러한 기존 도구들과 달리, 통신 데이터와 실행 시간을 결합하여 시스템의 효율성을 분석하는 프로파일링 관점의 접근 방식을 취한다는 점에서 차별성을 가진다.

## 🛠️ Methodology

### 1. 전체 시스템 아키텍처
AgentSpotter는 플랫폼 독립성을 최대화하기 위해 다음과 같은 구조를 가진다.
- **AgentSpotter Service**: 에이전트 플랫폼의 런타임 환경(RTE) 내에서 실행되며, 에이전트의 활동, 교환된 메시지, CPU 및 메모리 사용량 등의 데이터를 수집하여 Snapshot File(`.aspot`)에 저장한다.
- **AgentSpotter Station**: 저장된 스냅샷 파일을 읽어 쿼리 엔진을 통해 데이터를 분석하고, 이를 시각적으로 표현하는 애플리케이션이다.

### 2. Agent-Oriented Call Graph 모델
전통적인 Call Graph가 함수 호출 계층을 통해 실행 시간을 추적하듯, MAS Call Graph는 메시지 전달을 통해 처리 시간을 추적한다.

**에이전트 메시지 영향력(Agent Message Impact) 측정**:
에이전트 $A$가 에이전트 $B$에게 시간 $\alpha$에 메시지 $M_\alpha$를 보냈을 때, 이 메시지가 $B$에게 미치는 영향 $T_{M_{\alpha,B}}$는 $B$가 다음 메시지 $M_\Omega$를 받기 전까지 수행한 모든 계산 시간의 합으로 정의한다.

$$ T_{M_{\alpha,B}} = \sum_{t=\alpha}^{\Omega} b_t $$

여기서 $b_t$는 시간 $t$에서의 에이전트 $B$의 활동 지속 시간이다. 이를 확장하여 다음과 같은 지표들을 산출한다.

- **에이전트 $x$가 $y$에게 미친 총 영향력**:
$$ T_{x,y} = \sum_{m=1}^{M} \sum_{t=\alpha_m}^{\Omega_m} b_t $$
- **에이전트 $x$가 시스템 전체에 미친 총 영향력**:
$$ T_x = \sum_{a=1}^{N} T_{x,a} $$
- **세션 $S$의 총 영향력**:
$$ T_S = \sum_{k=1}^{K} T_k $$

또한, 세션의 전체 활동 시간 $A_S$는 총 영향력 $T_S$에 각 에이전트가 첫 메시지를 받기 전까지 수행한 초기 활동 시간을 더하여 계산한다.

### 3. 시각화 사양 및 인터페이스
Call Graph는 다음과 같은 고정 깊이(Fixed-depth)의 트리 구조로 표현된다:
**$\text{Session} \rightarrow \text{Emitter Agent} \rightarrow \text{Receiver Agent} \rightarrow \text{FIPA ACL Content} \rightarrow \text{Message}$**

각 노드에는 다음과 같은 정보가 표시된다:
- **Label**: 세션 정보, 에이전트 ID, FIPA ACL 수행문(Performative) 및 내용, 타임스탬프 등.
- **Total impact time**: 해당 노드 하위 자식들의 영향력 합계.
- **$\%$ parent time / $\%$ session time**: 부모 노드 및 전체 세션 대비 해당 노드의 영향력 비율.

특히, 사용자는 트리 계층을 유연하게 변경할 수 있다. 예를 들어, 수신자(Receiver)를 송신자(Emitter)보다 상위에 배치함으로써 특정 에이전트가 시스템의 병목 지점이 되고 있는지(부하 불균형)를 분석할 수 있다.

## 📊 Results

### 1. 실험 설정
제안된 도구의 유효성을 검증하기 위해 Overseer 에이전트(작업 요청)와 Worker 에이전트(작업 수행)로 구성된 벤치마크 애플리케이션을 구축하였다. Overseer는 Worker에게 소, 중, 대 규모의 작업을 요청하며, Worker는 과부하 상태일 경우 요청을 거부할 수 있다.

### 2. 주요 결과 및 분석
- **부하 불균형(Load Imbalance) 발견**: 분석 결과, `master1`과 `master2` 두 Overseer 에이전트의 영향력이 크게 달랐다. `master2`의 메시지 영향력은 전체 세션의 $20.4\%$에 불과했다.
- **상세 원인 분석**: Call Graph를 통해 확인한 결과, `master2`가 보낸 `pleaseDoThing(20)` 요청의 상당수가 $0.0$의 영향력을 가졌다. 이는 Worker 에이전트가 과부하 상태에서 `master2`의 요청을 무시(Ignore)했음을 의미한다. 반면 `master1`의 동일 요청은 정상적으로 처리되어 $8.8\%$의 영향력을 기록했다.
- **병목 지점(Bottleneck) 식별**: 메시지 내용에 따라 영향력을 비교했을 때, `pleaseDoThing(1)`보다 `pleaseDoThing(20)`의 영향력이 월등히 높게 나타났다. 이는 요청된 작업의 크기가 클수록 처리 시간이 증가한다는 직관과 일치하며, 만약 예상치 못한 메시지에서 높은 영향력이 발견된다면 해당 코드가 효율성 저하의 원인임을 빠르게 파악할 수 있다.

## 🧠 Insights & Discussion

### 1. 강점
본 연구는 MAS의 복잡한 통신 구조를 계층적 트리 형태로 단순화하여, 개발자가 코드 전체를 일일이 분석하지 않고도 시스템의 성능 병목과 부하 불균형을 시각적으로 즉각 파악할 수 있게 한다. 특히 비전문가나 테스터도 시스템의 이상 동작을 식별할 수 있다는 점이 큰 장점이다.

### 2. 한계 및 가정
본 모델은 **"에이전트의 행동은 수신한 메시지와 직접적으로 연관되어 있다"**는 단순한 가정을 바탕으로 한다. 그러나 다음과 같은 한계가 존재한다.
- **인과관계의 불명확성**: 에이전트는 환경의 변화(Percept)나 내부의 신념(Belief) 변화에 의해 행동할 수 있으며, 이는 ACL 메시지와 무관할 수 있다.
- **처리 시간의 중첩**: 하나의 메시지에 대한 처리가 완료되기 전에 다음 메시지를 수신하는 경우, 영향력 측정의 정확도가 떨어진다.

### 3. 비판적 해석
논문에서 언급했듯이, 내부 추론 프로세스를 추적하는 것이 가장 정확하겠지만 이는 플랫폼 의존성이 너무 높아져 AgentSpotter의 설계 목표인 '플랫폼 불가지론(Platform-agnostic)'에 위배된다. 따라서 현재의 근사치 측정 방식은 완벽하지는 않으나, 유사한 유형의 메시지들이 일관되게 높은 영향력을 보인다면 통계적으로 인과관계가 있을 가능성이 높으므로 실용적인 가치가 있다고 판단된다.

## 📌 TL;DR

본 논문은 Multi Agent System(MAS)의 성능 분석을 위해 전통적인 Call Graph 개념을 도입한 **AgentSpotter의 Call Graph View**를 제안한다. 에이전트 간의 메시지 전달을 '영향력(Impact)'이라는 지표로 정량화하여 트리 구조로 시각화함으로써, 시스템의 병목 지점과 부하 불균형을 효율적으로 찾아낼 수 있게 한다. 비록 메시지와 행동 간의 엄격한 인과관계를 증명하는 데는 한계가 있으나, MAS 개발 프로세스에서 성능 최적화를 위한 매우 유용한 진단 도구가 될 것으로 기대된다.
