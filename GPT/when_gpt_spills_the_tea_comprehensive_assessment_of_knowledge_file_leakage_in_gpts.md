# When GPT Spills the Tea: Comprehensive Assessment of Knowledge File Leakage in GPTs

Xinyue Shen, Yun Shen, Michael Backes, Yang Zhang (2025)

## 🧩 Problem to Solve

본 논문은 OpenAI의 GPTs와 같이 사용자가 맞춤형 지식 파일(Knowledge Files)을 업로드하여 성능을 향상시킬 수 있는 LLM 에이전트에서 발생하는 지식 파일 유출 문제를 다룬다. GPT 빌더가 업로드한 텍스트북, 의료 기록, 내부 문서 등의 지식 파일은 모델의 응답 품질을 높이는 핵심 요소이지만, 동시에 민감한 정보나 저작권 보호 자료를 포함하고 있어 보안상 매우 중요하다.

기존 연구들은 주로 적대적 프롬프트(Adversarial Prompts)를 통해 모델이 내용을 출력하게 만드는 머신러닝 관점의 유출에 집중하였다. 그러나 GPTs는 클라이언트, 서버, 데이터베이스를 거치는 복잡한 데이터 흐름을 가진 웹 애플리케이션의 형태를 띠고 있다. 따라서 단순한 프롬프트 주입 외에 데이터 공급망(Data Supply Chain) 전체에서 발생할 수 있는 추가적인 유출 경로가 존재할 가능성이 크며, 이를 체계적으로 분석할 필요가 있다. 본 논문의 목표는 DSPM(Data Security Posture Management) 프레임워크를 적용하여 GPTs의 지식 파일 유출 리스크를 포괄적으로 평가하고 이를 방어하기 위한 방안을 제시하는 것이다.

## ✨ Key Contributions

본 연구의 핵심 기여는 GPT 데이터 공급망 전반에 걸친 지식 파일 유출을 평가하기 위해 DSPM에서 영감을 얻은 새로운 리스크 평가 워크플로우를 제안하고 적용했다는 점이다. 특히, 단순한 텍스트 출력을 넘어 원본 파일 자체를 다운로드할 수 있게 하는 권한 상승(Privilege Escalation) 취약점을 발견하였다.

연구팀은 Built-in 도구인 Code Interpreter를 악용하여 원래는 접근이 제한된 원본 지식 파일을 직접 다운로드할 수 있는 경로를 찾아냈으며, 이는 기존의 프롬프트 기반 공격보다 훨씬 치명적인 위협임을 입증하였다. 또한, 대규모 메타데이터 분석을 통해 실제 유출된 파일 중 상당수가 주요 출판사의 저작권 자료나 기업 내부 기밀 문서임을 밝혀내어 해당 취약점의 실질적인 위험성을 경고하였다.

## 📎 Related Works

기존의 관련 연구들은 주로 프롬프트 주입(Prompt Injection) 공격을 통해 시스템 프롬프트나 지식 파일의 내용을 유출시키는 방법론에 집중하였다. 예를 들어, Yu et al. [36], Su et al. [31], Zhang et al. [40] 등의 연구는 적대적 프롬프트를 사용하여 모델이 지식 파일의 이름을 말하게 하거나 내용을 일부 출력하도록 유도하는 방식을 취하였다.

하지만 본 논문은 다음과 같은 차별점을 가진다. 첫째, 공격 표면을 단순히 모델의 입력(Prompt)으로 한정 짓지 않고, API 메타데이터, 웹소켓(WebSocket) 통신 흐름(Flows), 클라이언트 응답(Responses) 등 웹 애플리케이션 아키텍처 전체로 확장하였다. 둘째, 기존 연구들이 모델의 환각(Hallucination) 가능성으로 인해 유출 결과의 실제 정확성을 검증하기 어려웠던 반면, 본 연구는 여러 경로에서 수집한 데이터를 교차 검증하여 확실한 Ground Truth를 구축하고 유출의 정확도를 정량적으로 평가하였다.

## 🛠️ Methodology

본 논문은 DSPM의 데이터 중심 접근 방식을 따라 다음과 같은 4단계 워크플로우를 통해 리스크를 평가한다.

### 1. 데이터 발견 (Data Discovery)

지식 파일이 유출될 수 있는 세 가지 주요 데이터 소스를 정의한다.

- **Metadata**: GPT Store에서 API를 통해 제공되는 JSON 형태의 정보.
- **Flows**: 사용자와 서버 간의 웹소켓 통신을 통해 전송되는 구조화된 메시지.
- **Response**: 최종적으로 사용자 인터페이스에 렌더링되는 텍스트 응답.

### 2. 데이터 분류 (Data Classification)

유출된 데이터의 민감도에 따라 지식 파일 정보를 7가지 차원으로 분류한다: $\text{ID}, \text{type}, \text{count}, \text{size}, \text{title}, \text{content}, \text{original files}$. 이 중 title, content, original files는 가장 민감한 정보로 간주한다.

### 3. 리스크 평가 (Risk Assessment)

다섯 가지 유출 벡터(Leakage Vector)를 중심으로 취약점을 분석한다.

- **Metadata & GPT Initialization**: 서버 측 필터링 부족으로 인해 API 응답이나 초기화 흐름에서 파일의 ID, 크기, 제목 등이 노출되는 Excessive Information Exposure 문제가 발생한다.
- **Retrieval**: `myfiles_browser`라는 내장 도구가 시맨틱 검색을 수행할 때, 파일 크기가 작은 순서대로 우선적으로 검색되어 내용이 Flows에 노출되는 특성이 있다. 누적 크기가 약 $100\text{K}$ tokens를 초과하면 이후 파일은 제외되는 경향을 보인다.
- **Sandboxed Execution Environment (SEE)**: Code Interpreter 활성화 시, 파일이 `/mnt/data` 디렉토리에 저장된다. 기본적으로는 403 에러로 접근이 제한되지만, "파일을 새 파일로 복사하여 다운로드 링크를 생성하라"는 프롬프트를 통해 권한 상승을 유도하여 원본 파일을 다운로드할 수 있다. 이는 전형적인 Broken Access Control 취약점에 해당한다.
- **Prompt**: 프롬프트를 통해 내용을 유출시키는 기존 방식으로, 모델의 환각으로 인해 정확도가 낮게 나타난다.

### 4. 완화 (Mitigation)

분석 결과를 바탕으로 빌더와 플랫폼 제공자를 위한 해결책을 제시한다.

- **빌더**: Code Interpreter 비활성화, 명시적인 방어 프롬프트(Defense Prompts) 작성, 또는 실제 파일 업로드 전 무의미한 데이터를 채운 '더미 파일'을 먼저 업로드하여 `myfiles_browser`의 검색 우선순위를 조작하는 방법 등을 제안한다.
- **플랫폼(OpenAI)**: 클라이언트 측 필터링이 아닌 서버 측에서 데이터를 엄격히 필터링하도록 API를 재설계해야 함을 강조한다.

## 📊 Results

### 실험 설정

연구팀은 GPT Store의 $651,022$개 GPT 메타데이터, $1,466$개 GPT에서 추출한 $11,820$개의 Flows 및 $1,466$개의 Responses를 분석하였다.

### 주요 정량적 결과

- **SEE 권한 상승 취약점**: Code Interpreter가 활성화된 GPT의 경우, 원본 파일 다운로드 성공률이 $95.95\%$에 달하였다. 반면, 비활성화된 경우 유출률은 $0.00\%$였다.
- **저작권 침해 분석**: 유출된 원본 PDF 파일 $566$개를 수동 분석한 결과, $28.80\%$($163$개)가 명백한 저작권 침해 파일이었다. 여기에는 Springer, Elsevier, O’Reilly와 같은 주요 출판사의 도서와 시가 $4$억 달러 규모 상장사의 내부 연례 보고서 등이 포함되었다.
- **프롬프트 기반 유출 정확도**: Code Interpreter 활성화 시 프롬프트 기반 추출의 정확도는 $0.842$였으나, 비활성화 시 $0.654$로 크게 떨어졌다. 이는 모델의 환각 현상이 유출 결과에 영향을 미침을 시사한다.

### 정성적 결과

`myfiles_browser`를 통한 유출은 파일 크기에 매우 의존적이며, 작은 파일들이 우선적으로 유출되는 패턴이 확인되었다. 또한, 일부 빌더가 적용한 방어 프롬프트는 특정 조건에서만 작동하거나 완전히 무력화되는 경우가 많아, 단순한 지침보다는 시스템 레벨의 제어가 필수적임을 보여주었다.

## 🧠 Insights & Discussion

본 논문은 GPTs의 보안 모델이 '사용자 인터페이스'라는 얇은 막에 의존하고 있으며, 실제 데이터 공급망의 하부 구조(API, WebSocket)에서는 심각한 설계 결함이 존재함을 드러냈다. 특히 Code Interpreter를 통한 권한 상승 취약점은 단순한 정보 유출을 넘어 원본 데이터의 무단 탈취가 가능하다는 점에서 매우 위험하다.

저자들은 이 취약점이 OWASP의 API 보안 위험 중 'Excessive Data Exposure'와 'Broken Access Control'에 해당한다고 분석한다. 이는 OpenAI가 편의성을 위해 서버에서 데이터를 충분히 필터링하지 않고 클라이언트에 전송한 뒤, 클라이언트 단에서 보여줄 부분만 선택하는 방식을 취했기 때문으로 해석된다.

비판적 관점에서 볼 때, 빌더가 적용할 수 있는 '더미 파일 업로드' 같은 우회책은 임시방편일 뿐이며, 근본적인 해결책은 플랫폼 제공자가 데이터 접근 제어 리스트(ACL)를 서버 사이드에서 엄격하게 관리하는 것뿐이다. 또한, 저작권 보호 자료가 이토록 쉽게 유출된다는 점은 향후 LLM 플랫폼의 법적 책임 및 저작권 관리 체계에 큰 논쟁을 불러일으킬 가능성이 높다.

## 📌 TL;DR

본 연구는 GPTs의 지식 파일 유출 위험을 DSPM 프레임워크를 통해 체계적으로 분석하여, 메타데이터, 초기화 흐름, 검색 도구, 샌드박스 환경(SEE), 프롬프트라는 5가지 유출 경로를 식별하였다. 특히 Code Interpreter를 이용해 원본 파일을 직접 다운로드할 수 있는 심각한 권한 상승 취약점을 발견하였으며(성공률 $95.95\%$), 이를 통해 유출된 파일의 약 $28.8\%$가 저작권 침해 자료임을 확인하였다. 이 연구는 LLM 에이전트의 보안이 단순히 프롬프트 방어를 넘어 전체 데이터 공급망의 아키텍처 설계 수준에서 다뤄져야 함을 시사하며, 향후 LLM 플랫폼의 보안 표준 설정에 중요한 근거를 제공한다.
