# RL in the Wild: Characterizing RLVR Training in LLM Deployment

Jiecheng Zhou, Qinghao Hu, Yuyang Jin, Zerui Wang, Peng Sun, Yuzhe Gu, Wenwei Zhang, Mingshu Zhai, Xingcheng Zhang, Weiming Zhang (2025)

## 🧩 Problem to Solve

최근 대규모 언어 모델(LLM)의 추론 및 이해 능력을 향상시키기 위해 검증 가능한 보상을 사용하는 강화학습인 RLVR(Reinforcement Learning with Verifiable Rewards)이 급격히 부상하고 있다. 그러나 RLVR은 복잡한 데이터 흐름과 다양한 태스크 특성으로 인해 학습 시스템 구축에 상당한 어려움을 야기한다.

기존의 LLM 워크로드 분석은 주로 사전 학습(Pretraining)이나 서빙(Serving)에 집중되어 있었으며, RLVR 학습 시스템의 관점에서 시스템적 도전 과제를 분석한 연구는 매우 부족한 실정이다. 특히 RLVR은 일반적인 학습과 달리 다음과 같은 고유한 특성을 가진다:

1. **긴 스텝 간격(Longer step span):** 한 번의 학습 스텝이 수 분에서 한 시간 이상 소요될 수 있다.
2. **복잡한 워크플로우(Complicated workflow):** Rollout, Inference, Training의 여러 단계와 다수의 모델이 상호작용한다.
3. **예측 불가능한 시퀀스 길이(Unknown sequence length):** 모델의 생성 결과에 따라 워크로드가 동적으로 변하며, 이는 병렬화 전략의 비효율성으로 이어진다.
4. **복잡한 소프트웨어 스택(Complex software stack):** 학습 및 추론 프레임워크뿐만 아니라 샌드박스(Sandbox)와 같은 외부 환경과의 상호작용 소프트웨어가 통합되어야 한다.

본 논문의 목표는 실제 배포 환경에서 RLVR 태스크의 워크로드를 정밀하게 분석하여 시스템적 병목 지점을 식별하고, 이를 기반으로 현실적인 워크로드를 재현할 수 있는 PolyTrace 벤치마크 스위트를 제안하는 것이다.

## ✨ Key Contributions

본 논문의 핵심 기여는 RLVR 학습 시스템에 대한 최초의 종합적인 워크로드 분석을 수행하고, 이를 통해 시스템 최적화를 위한 구체적인 통찰을 제공한 점이다.

- **RLVR 워크로드 특성 분석:** 다양한 RL 태스크(수학, 코딩, 멀티모달 이해, 도구 사용 등)에서 입력 및 출력 시퀀스 길이의 분포와 그 변동 추이를 정량적으로 분석하였다.
- **시스템 병목 지점 식별:** GPU 유휴 상태(Idling)를 유발하는 롱테일(Long-tail) 분포, 비효율적인 데이터 전송 메커니즘, 정적 병렬화 전략의 한계, 그리고 CPU 오버헤드가 큰 추론 백엔드의 문제를 밝혀냈다.
- **PolyTrace 벤치마크 스위트 제안:** 실제 수집한 트레이스 데이터를 기반으로 현실적인 RL 워크로드를 생성하고 시뮬레이션할 수 있는 벤치마크 도구를 개발하여, 연구자들이 실제 모델을 학습시키지 않고도 시스템 최적화를 검증할 수 있게 하였다.

## 📎 Related Works

논문에서는 LLM 학습 및 서빙 시스템에 대한 기존 연구들을 언급하며 RLVR과의 차이점을 설명한다.

- **기존 워크로드 분석:** Pretraining(MegaScale, ByteScale)이나 Serving(ServeGen, KV Cache in the Wild)에 대한 분석은 존재하지만, 이들은 데이터가 정적이거나 요청 패턴이 RLVR과 근본적으로 다르다.
- **RL 학습 프레임워크:** Verl, RealHF 등 최신 RL 프레임워크들이 등장하여 자원 할당 및 파이프라이닝 최적화를 시도하고 있으나, 실제 배포 환경에서의 정밀한 워크로드 특성에 기반한 시스템 최적화 연구는 부족하다.
- **비동기 RL 학습:** 최근 AsyncFlow 등의 연구를 통해 Rollout과 Training 단계를 분리하여 GPU 유휴 시간을 줄이려는 시도가 있으나, 본 논문은 이러한 시스템 설계가 실제 워크로드의 동적 특성에 의해 어떻게 영향을 받는지를 분석한다.

## 🛠️ Methodology

본 논문은 특정 알고리즘을 제안하는 것이 아니라, 대규모 RLVR 학습 시스템의 동작을 분석하는 방법론을 제시한다.

### 1. RLVR 학습 파이프라인 구조

RLVR 학습은 크게 세 가지 단계로 구성되며, 각 단계는 서로 다른 컴퓨팅 특성을 가진다:

- **Rollout:** Actor 모델이 환경과 상호작용하며 텍스트를 생성하는 단계이다. 주로 vLLM, SGLang과 같은 추론 엔진이 담당하며, 메모리 대역폭에 제한을 받는(Memory-bound) 특성을 보인다.
- **Inference:** 생성된 결과에 대해 Reward 모델이나 Judge 모델이 보상을 계산하는 단계이다.
- **Training:** 계산된 보상을 바탕으로 Actor 모델(및 Critic 모델)의 파라미터를 업데이트하는 단계이다. Megatron, FSDP 등의 학습 프레임워크가 사용되며, 계산 집약적인(Compute-bound) 특성을 보인다.

### 2. 분석 대상 및 환경

- **대상 태스크:** 이미지 이해, 수학, 도구 사용(Tool-use), 코딩, 비디오 이해, 검색 등 7가지 대표 태스크를 분석하였다.
- **하드웨어:** NVIDIA A800 및 H800 GPU 클러스터를 사용하여 대규모 학습 환경을 구축하였다.

### 3. PolyTrace 벤치마크 설계

실제 워크로드를 재현하기 위해 다음과 같은 구성 요소를 설계하였다:

- **Tool use latency sampler:** 외부 API 호출 시 발생하는 변동성이 큰 지연 시간을 시뮬레이션하기 위해 실제 수집된 지연 시간 분포에서 무작위 샘플링을 수행한다.
- **Workload sampler and generator:** 수집된 트레이스에서 입력/출력 길이, 대화 턴 수 등의 메타데이터를 샘플링하여 더미 데이터(Dummy inputs)를 생성한다.
- **Execution Control:** `ignore_eos_token=True` 및 `max_output_len` 설정을 통해 추론 엔진이 실제 데이터 생성 시와 동일한 연산 시간을 소모하도록 강제한다.

## 📊 Results

### 1. 시퀀스 길이 분포 분석

- **태스크별 극명한 차이:** 수학 및 코딩 태스크는 매우 긴 출력 시퀀스(평균 9,000개 이상의 토큰)를 생성하는 반면, 검색 태스크는 매우 짧은 출력을 보인다. 반대로 비디오 이해 태스크는 입력 시퀀스 길이가 매우 길다.
- **롱테일 분포:** 특히 도구 사용 태스크의 경우 입력 길이의 90th percentile과 최대값의 차이가 최대 18배에 달하는 심한 롱테일 분포를 보인다.
- **동적 변동성:** 학습이 진행됨에 따라 모델의 능력이 향상되면서 수학/코딩 태스크의 출력 길이는 점차 증가하는 경향을 보였으나, 검색 태스크는 도구 사용 효율이 높아지며 오히려 출력 길이가 짧아지는 양상을 보였다.

### 2. 시스템 성능 분석

- **TGS(Tokens per GPU per Second) 변동성:** 이미지 이해 태스크의 경우 최저 0.8 TGS에서 최고 341 TGS까지 성능이 약 400배 이상 격차가 발생한다. 이는 시퀀스 길이의 불균형과 부적절한 병렬화 전략 때문이다.
- **시간 소요 분석:** 대부분의 태스크에서 Rollout 단계가 전체 시간의 대부분을 차지하며, 특히 추론 엔진의 스케줄링 오버헤드(CPU overhead)가 상당한 비중을 차지함을 확인하였다.
- **확장성(Scalability) 문제:** GPU 수를 늘려도 성능 향상이 미비하거나 오히려 저하되는 'Negative Scaling' 현상이 관찰되었다. 이는 소수의 GPU가 매우 긴 시퀀스를 처리하는 동안 나머지 GPU들이 대기하는 GPU Idling 문제와 통신 오버헤드 증가 때문이다.

### 3. 벤치마크 정확도

PolyTrace 벤치마크 스위트를 사용하여 비디오 이해 태스크의 지연 시간을 측정한 결과, 실제 실행 시간($511.44\text{s}$) 대비 시뮬레이션 시간($484.48\text{s}$)의 오차가 약 $5.3\%$로, $94.7\%$의 높은 정확도를 보였다.

## 🧠 Insights & Discussion

### 1. 주요 통찰 및 강점

- **워크로드 인식 스케줄링의 필요성:** 단순한 FCFS(First Come First Serve) 방식의 스케줄링으로는 RLVR의 극심한 시퀀스 길이 불균형을 해결할 수 없으며, 프롬프트 단위의 스케줄링이나 사전 샘플링을 통한 길이 예측 기반 스케줄링이 필요하다.
- **데이터 전송 구조의 개선:** 현재 많은 프레임워크가 채택하고 있는 Single-controller 구조는 데이터량이 많은 멀티모달/롱컨텍스트 태스크에서 직렬화/역직렬화 오버헤드로 인한 병목 현상과 CPU OOM을 유발한다.
- **동적 병렬화 전략:** 정적인 Sequence Parallel 설정은 짧은 시퀀스에서는 오히려 통신 오버헤드를 증가시킨다. 따라서 시퀀스 길이에 따라 병렬화 전략을 실시간으로 변경하는 Online sequence-length-aware 전략이 필수적이다.

### 2. 한계 및 비판적 해석

- **알고리즘의 급격한 변화:** RLVR 알고리즘이 매우 빠르게 발전하고 있어, 본 논문에서 분석한 태스크 외에 새로운 유형의 워크로드가 계속 등장할 가능성이 크다.
- **결함 허용(Fault Tolerance) 부재:** 시스템 성능 분석에 집중하느라, 대규모 학습에서 필수적인 체크포인트 저장 및 복구, 노드 장애 대응 등의 안정성 측면은 다루지 않았다.
- **시뮬레이터의 의존성:** PolyTrace는 실제 추론 엔진을 실행하여 시간을 측정하므로, 완전한 수학적 시뮬레이터보다는 자원 소모가 크다.

## 📌 TL;DR

본 논문은 LLM의 추론 능력을 높이는 RLVR 학습 과정에서 발생하는 시스템적 병목 현상을 실제 배포 환경의 데이터를 통해 분석한 연구이다. 분석 결과, **극심한 시퀀스 길이의 불균형(Long-tail distribution)**이 GPU 유휴 상태를 유발하고, **단일 컨트롤러 기반의 데이터 전송** 및 **정적 병렬화 전략**이 성능을 저하시키는 주요 원인임을 밝혀냈다. 이를 해결하기 위한 기반으로 현실적인 워크로드를 재현하는 **PolyTrace 벤치마크**를 제안하였으며, 이는 향후 RLVR 전용 효율적인 학습 시스템 및 동적 스케줄링 알고리즘 연구에 중요한 기초 자료가 될 것으로 기대된다.
